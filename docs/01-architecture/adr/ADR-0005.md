# ADR-0005

# Adopt Event-Driven Architecture

> **Status:** Accepted
>
> **Date:** 2026-07-01
>
> **Deciders:** Atlas Studio Architecture Team

---

# Context

Atlas Studio contains multiple independent subsystems.

Examples include:

- Rendering
- OCR
- Translation
- Autosave
- History
- User Interface
- Plugins

These subsystems need to react to document changes without becoming tightly coupled to each other.

Direct method calls between subsystems would introduce unnecessary dependencies and reduce extensibility.

---

# Problem

A document modification may require multiple follow-up actions.

Example:

```
Bubble translated

↓

Update renderer

↓

Update history

↓

Autosave

↓

Notify plugins

↓

Refresh UI
```

If the command handler invokes each subsystem directly, responsibilities become tightly coupled.

---

# Decision

Atlas Studio adopts an Event-Driven Architecture.

Commands modify the document.

Events announce completed changes.

Subscribers react independently.

Commands never depend on subscribers.

---

# Architecture

```text
User
    │
    ▼
Command
    │
    ▼
Command Handler
    │
    ▼
Document Mutation
    │
    ▼
Domain Event
    │
    ▼
Event Bus
    │
    ├────────► History
    │
    ├────────► Autosave
    │
    ├────────► Renderer
    │
    ├────────► Plugins
    │
    ├────────► UI
    │
    └────────► Analytics
```

---

# Domain Events

Domain events represent facts that have already happened.

Examples

```
ProjectCreated

PageAdded

PageRemoved

LayerMoved

BubbleCreated

BubbleDeleted

BubbleMoved

TranslationUpdated

ProjectSaved
```

Events use the past tense.

---

# Event Bus

All events are dispatched through the Event Bus.

Responsibilities

- Publish events
- Register subscribers
- Dispatch events
- Preserve execution order
- Support asynchronous subscribers

The Event Bus owns delivery.

---

# Event Publisher

Command handlers publish events after a successful mutation.

Example

```text
Command

↓

Mutation

↓

Publish Event
```

Events are never published before document mutation succeeds.

---

# Event Subscribers

Subscribers react to events.

Examples

```
HistorySubscriber

AutosaveSubscriber

RenderSubscriber

PluginSubscriber

AnalyticsSubscriber
```

Subscribers should remain independent.

Subscribers should not communicate with one another.

---

# Event Ordering

Events are dispatched in the same order they are published.

Ordering must be deterministic.

---

# Event Delivery

The default delivery mechanism is synchronous.

Future implementations may support asynchronous dispatch without changing the public API.

---

# Event Payload

Events should contain only the information required by subscribers.

Good

```
BubbleId

ProjectId

TranslationId
```

Avoid embedding entire document objects.

Subscribers should query additional data if necessary.

---

# Event Immutability

Events are immutable.

Subscribers must never modify event payloads.

---

# Event Naming

Use:

```
Noun + Past Tense
```

Examples

```
ProjectCreated

BubbleRemoved

TranslationUpdated
```

Avoid

```
CreateProject

RemoveBubble

UpdateTranslation
```

---

# Event Scope

Events describe domain changes.

Events must not represent UI actions.

Good

```
PageAdded
```

Bad

```
ButtonClicked
```

UI events belong to the application layer.

---

# Error Handling

Subscriber failures must not corrupt document state.

The mutation has already completed.

Errors should be handled according to the configured event dispatch policy.

---

# Package Responsibilities

atlas-events

Responsible for

- Event Bus
- Event Dispatcher
- Event Subscription
- Event Middleware

atlas-command

Responsible for

- Publishing events

atlas-document

Responsible for

- No event handling

atlas-core

Responsible for

- Business subscribers

---

# Plugin Integration

Plugins subscribe through the public Event Bus.

Plugins never intercept command execution.

Plugins react after document changes occur.

---

# AI Integration

AI services subscribe to events like any other subsystem.

Examples

```
TranslationUpdated

↓

Quality Checker

↓

Suggestion Generator

↓

Analytics
```

AI should never bypass the Event Bus.

---

# Rationale

Event-Driven Architecture provides:

- Loose coupling
- Extensibility
- Independent subsystems
- Better testing
- Plugin support
- Future asynchronous processing

without increasing package dependencies.

---

# Consequences

## Positive

- Decoupled architecture.
- Easier feature expansion.
- Independent subscribers.
- Cleaner package boundaries.
- Better plugin support.
- Better observability.

## Negative

- Event flow is less explicit than direct method calls.
- Debugging requires tracing published events.
- Event ordering must be carefully managed.

---

# Alternatives Considered

## Direct Service Calls

Rejected.

Reason:

Introduces tight coupling between subsystems.

---

## Observer Pattern Inside Documents

Rejected.

Reason:

Would make the document model responsible for event management.

---

## Global Callback Registry

Rejected.

Reason:

Poor encapsulation and difficult lifecycle management.

---

# Compliance

Every domain event must satisfy:

- [ ] Represents a completed fact.
- [ ] Uses past-tense naming.
- [ ] Is immutable.
- [ ] Is published after a successful mutation.
- [ ] Contains only required payload.
- [ ] Is dispatched through the Event Bus.
- [ ] Does not expose internal implementation details.

Architectural exceptions require a new ADR.
