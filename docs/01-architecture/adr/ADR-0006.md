# ADR-0006

# Adopt a Plugin Architecture

> **Status:** Accepted
>
> **Date:** 2026-07-01
>
> **Deciders:** Atlas Studio Architecture Team

---

# Context

Atlas Studio is designed to be extensible.

Users and third-party developers should be able to add new functionality without modifying the core source code.

Examples include:

- OCR providers
- Translation providers
- AI providers
- Importers
- Exporters
- File formats
- Automation tools
- Editor tools

Without a plugin architecture, every new feature would require modifying the core packages.

---

# Problem

Atlas Studio will continue to grow over time.

Different users may require:

- Different translation providers
- Different OCR engines
- Different export formats
- Different automation workflows

Embedding all features into atlas-core would create:

- Large package size
- Tight coupling
- Difficult maintenance
- Slow releases

---

# Decision

Atlas Studio adopts a Plugin Architecture.

Core packages define extension points.

Plugins register themselves through a Plugin Manager.

Plugins communicate only through stable public APIs.

Plugins must never modify internal implementation details.

---

# Architecture

```text
                Atlas Studio

                       │

                Plugin Manager

      ┌──────────┼──────────┐

      ▼          ▼          ▼

 Translation   Export     OCR

   Plugin      Plugin    Plugin

      ▼          ▼          ▼

   Public APIs only

      ▼

atlas-core

atlas-command

atlas-events

atlas-document

atlas-types
```

---

# Plugin Principles

Plugins:

- Extend functionality.
- Never replace core architecture.
- Depend only on public APIs.
- Must be independently installable.
- Must be independently removable.

---

# Extension Points

Core packages expose extension points.

Examples

```
TranslationProvider

OCRProvider

Exporter

Importer

CommandProvider

EventSubscriber

AIProvider
```

Plugins register implementations.

---

# Plugin Lifecycle

Every plugin follows the same lifecycle.

```text
Install

↓

Load

↓

Register

↓

Activate

↓

Execute

↓

Deactivate

↓

Unload

↓

Remove
```

The Plugin Manager controls the lifecycle.

---

# Plugin Manifest

Every plugin must provide a manifest.

Example

```json
{
  "id": "google-translate",
  "name": "Google Translate",
  "version": "1.0.0",
  "author": "Atlas Studio",
  "description": "Translation provider.",
  "entry": "./index.js",
  "engines": {
    "atlas": "^1.0.0"
  }
}
```

The manifest is the plugin's identity.

---

# Plugin Registration

Plugins register capabilities during activation.

Example

```text
Plugin

↓

Plugin Manager

↓

Register Provider

↓

Provider Registry
```

Plugins should not self-register outside the Plugin Manager.

---

# Plugin Communication

Plugins communicate through:

- Public APIs
- Commands
- Events
- Extension Points

Plugins never communicate through internal classes.

---

# Plugin Capabilities

Plugins may provide:

- Translation providers
- OCR providers
- AI providers
- Importers
- Exporters
- File readers
- File writers
- Editor tools
- Automation
- Analytics

Capabilities are declared explicitly.

---

# Plugin Isolation

Plugins should be isolated.

One plugin failure must not crash Atlas Studio.

Plugin exceptions are contained by the Plugin Manager.

---

# Plugin Dependencies

Plugins may depend on:

- atlas-types
- atlas-document
- atlas-command
- atlas-events
- atlas-core

Plugins must not depend on:

- Internal source directories
- Application internals
- Other plugins (unless explicitly declared and supported)

---

# Plugin Permissions

Plugins should request capabilities.

Examples

```
Translation

OCR

Filesystem

Network

Clipboard
```

The runtime may allow or deny permissions.

---

# Plugin Versioning

Plugins declare supported Atlas Studio versions.

Example

```json
{
  "engines": {
    "atlas": "^1.2.0"
  }
}
```

Incompatible plugins should not be loaded.

---

# Plugin Discovery

Plugins may be discovered from:

```
plugins/

Installed packages

Marketplace

Development mode
```

Discovery is handled by the Plugin Manager.

---

# Plugin API

Plugins interact through stable interfaces.

Example

```ts
export interface Plugin {
  activate(context: PluginContext): Promise<void>;

  deactivate(): Promise<void>;
}
```

Activation should be lightweight.

---

# Plugin Context

PluginContext provides access to approved APIs.

Examples

```
Command Bus

Event Bus

Provider Registry

Configuration

Logger
```

PluginContext should not expose internal implementation details.

---

# Security

Plugins execute with limited privileges.

They must not:

- Modify private APIs.
- Access internal state directly.
- Bypass the Command Framework.
- Bypass the Event Bus.

---

# Package Responsibilities

atlas-core

Provides extension points.

atlas-command

Provides command registration.

atlas-events

Provides event subscription.

atlas-document

Provides document model.

Plugin Manager

Coordinates plugin lifecycle.

Plugins

Provide implementations.

---

# Rationale

The Plugin Architecture provides:

- Extensibility
- Independent feature development
- Cleaner core packages
- Faster experimentation
- Third-party ecosystem support

without increasing coupling.

---

# Consequences

## Positive

- Small core packages.
- Independent plugin releases.
- Better maintainability.
- Easier experimentation.
- Community extensions.
- Reduced coupling.

## Negative

- More runtime complexity.
- Plugin compatibility management.
- Version negotiation.
- Security considerations.

---

# Alternatives Considered

## Built-in Providers Only

Rejected.

Reason:

Every new provider requires modifying the core packages.

---

## Runtime Monkey Patching

Rejected.

Reason:

Unsafe, difficult to maintain, and incompatible with stable APIs.

---

## Direct Internal Access

Rejected.

Reason:

Breaks encapsulation and makes internal refactoring difficult.

---

# Compliance

Every plugin must satisfy:

- [ ] Has a valid manifest.
- [ ] Uses only public APIs.
- [ ] Registers through the Plugin Manager.
- [ ] Declares capabilities.
- [ ] Supports activation and deactivation.
- [ ] Handles failures gracefully.
- [ ] Does not access internal modules.
- [ ] Does not modify document state outside the Command Framework.

Architectural exceptions require a new ADR.
