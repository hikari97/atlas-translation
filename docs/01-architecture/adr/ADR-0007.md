# ADR-0007

# Adopt a Service Provider Architecture

> **Status:** Accepted
>
> **Date:** 2026-07-01
>
> **Deciders:** Atlas Studio Architecture Team

---

# Context

Atlas Studio integrates multiple external technologies and services.

Examples include:

- OCR Engines
- Translation Engines
- AI Models
- Importers
- Exporters
- File Systems
- Storage Providers
- Rendering Backends

Each capability may have multiple implementations.

For example:

Translation

- Google Translate
- DeepL
- OpenAI
- Gemini
- Ollama

OCR

- Tesseract
- PaddleOCR
- EasyOCR

Hardcoding implementations into atlas-core would tightly couple business logic to specific vendors.

---

# Problem

Business logic should not depend on vendor implementations.

For example, the translation pipeline should not know whether the translation comes from:

- Google Translate
- DeepL
- OpenAI
- Offline model

The pipeline should only request translation.

---

# Decision

Atlas Studio adopts a Service Provider Architecture.

Business services depend on provider interfaces.

Concrete providers are registered through Provider Registries.

Providers may originate from:

- Built-in packages
- Plugins
- Future extensions

---

# Architecture

```text
                 Application
                      │
                      ▼
             TranslationService
                      │
                      ▼
            TranslationProvider
                      │
      ┌───────────────┼───────────────┐
      ▼               ▼               ▼
Google Provider   DeepL Provider   OpenAI Provider
```

Business services remain vendor-independent.

---

# Service Layer

Services coordinate workflows.

Examples

```
TranslationService

OCRService

RenderService

ExportService

ImportService

ValidationService
```

Services contain business logic.

Services never communicate directly with vendor SDKs.

---

# Provider Layer

Providers implement external integrations.

Examples

```
GoogleTranslateProvider

DeepLProvider

OpenAIProvider

GeminiProvider

TesseractProvider
```

Providers encapsulate all vendor-specific behavior.

---

# Provider Registry

Every provider type owns a registry.

Examples

```
TranslationProviderRegistry

OCRProviderRegistry

ExporterRegistry

ImporterRegistry
```

Registries manage provider discovery and selection.

---

# Service Flow

```text
Application

↓

Command

↓

TranslationService

↓

TranslationProviderRegistry

↓

TranslationProvider

↓

Vendor API
```

Business services remain independent of vendor implementations.

---

# Provider Selection

Providers may be selected using:

- Configuration
- User preference
- Project settings
- Runtime capability
- Automatic fallback

Provider selection is handled by the registry.

---

# Multiple Providers

Multiple providers may coexist.

Example

```
Translation

↓

Google

↓

DeepL

↓

OpenAI

↓

Offline
```

The registry determines which provider should execute.

---

# Provider Priority

Providers may expose metadata.

Examples

```
Priority

Language Support

Capabilities

Cost

Offline Support

Availability
```

Registries may use this information when selecting providers.

---

# Capability Discovery

Providers explicitly declare capabilities.

Examples

```
SupportsBatchTranslation

SupportsStreaming

SupportsOfflineMode

SupportsImageOCR

SupportsVerticalText
```

Capabilities allow services to adapt automatically.

---

# Provider Lifecycle

Every provider follows the same lifecycle.

```text
Create

↓

Register

↓

Initialize

↓

Execute

↓

Dispose
```

The registry owns lifecycle management.

---

# Plugin Integration

Plugins may register providers.

Example

```
Plugin

↓

TranslationProvider

↓

Registry

↓

TranslationService
```

Plugins never modify service implementations.

---

# Error Handling

Provider failures are isolated.

The registry may:

- Retry
- Select another provider
- Report failure

Business services remain unaffected.

---

# Package Responsibilities

atlas-core

Responsible for:

- Business services
- Provider registries
- Workflow orchestration

Plugins

Responsible for:

- Provider implementations

atlas-document

Responsible only for document state.

atlas-command

Responsible only for command execution.

atlas-events

Responsible only for event dispatching.

---

# Public API

Consumers interact with services.

Good

```ts
await translationService.translate(bubble);
```

Consumers never instantiate providers directly.

---

# Dependency Rules

Services depend on provider interfaces.

Providers implement those interfaces.

Services never depend on vendor SDKs.

---

# Rationale

Service Provider Architecture provides:

- Vendor independence
- Extensibility
- Better testing
- Easier plugin support
- Runtime provider selection
- Reduced coupling

---

# Consequences

## Positive

- Replace providers without changing business logic.
- Support multiple providers simultaneously.
- Easier testing with mock providers.
- Better plugin ecosystem.
- Cleaner separation of concerns.

## Negative

- Additional abstraction.
- Registry management.
- Provider lifecycle management.

---

# Alternatives Considered

## Direct SDK Integration

Rejected.

Reason:

Business services become tightly coupled to vendor implementations.

---

## Static Provider Selection

Rejected.

Reason:

Prevents runtime flexibility and plugin extensibility.

---

## Service Locator

Rejected.

Reason:

Introduces hidden dependencies and makes testing more difficult.

---

# Compliance

Every provider must satisfy:

- [ ] Implements the appropriate provider interface.
- [ ] Registers through a provider registry.
- [ ] Encapsulates vendor-specific behavior.
- [ ] Exposes capabilities explicitly.
- [ ] Handles errors gracefully.
- [ ] Does not contain business workflow logic.
- [ ] Can be replaced without modifying services.

Architectural exceptions require a new ADR.
