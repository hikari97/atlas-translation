# ADR-0004

# Adopt the Command Pattern

> **Status:** Accepted
>
> **Date:** 2026-07-01
>
> **Deciders:** Atlas Studio Architecture Team

---

# Context

Atlas Studio is an interactive document editor.

Every user interaction modifies the document.

Examples include:

- Creating a project
- Opening a project
- Adding a page
- Deleting a page
- Creating a bubble
- Editing text
- Moving layers
- Translating text

Without a common execution model, every subsystem could modify the document differently.

This would make undo/redo, history, automation, plugins, and collaboration difficult to implement.

---

# Problem

Document modifications originate from many sources.

Examples:

- User Interface
- Keyboard Shortcuts
- AI
- Plugins
- Importers
- Scripts
- Automation

Allowing every subsystem to modify the document directly creates inconsistent behavior.

---

# Decision

Atlas Studio adopts the Command Pattern.

Every document modification must be represented as a Command.

Commands are executed through the Command Framework.

No subsystem may modify the document directly.

---

# Command Flow

```text
User Action
      │
      ▼
Command
      │
      ▼
Command Handler
      │
      ▼
Mutation Framework
      │
      ▼
Document Object Model
      │
      ▼
Domain Events
```

Every write operation follows this flow.

---

# Responsibilities

Commands describe an intention.

Examples

```
CreateProjectCommand

DeletePageCommand

RenameLayerCommand

MoveBubbleCommand

UpdateTranslationCommand
```

Commands do not contain business logic.

---

# Command Handlers

Business logic belongs inside handlers.

Examples

```
CreateProjectHandler

DeletePageHandler

MoveBubbleHandler

TranslateBubbleHandler
```

Handlers:

- Validate input.
- Coordinate dependencies.
- Apply document mutations.
- Produce domain events.
- Return execution results.

---

# Command Bus

Commands are dispatched through a Command Bus.

Example

```text
UI

↓

Command Bus

↓

Handler

↓

Document
```

Consumers never instantiate handlers directly.

---

# Undo / Redo

Undo and Redo operate on executed commands.

Every reversible command should provide:

- Execute
- Undo

Redo simply executes the command again.

---

# Transactions

Multiple commands may execute inside a transaction.

Example

```
Import Manga

↓

Create Project

↓

Create Pages

↓

Create Layers

↓

Create Bubbles

↓

Commit
```

If one command fails, the transaction may roll back.

---

# Composite Commands

Multiple commands may be grouped.

Example

```
ImportCBZCommand

↓

CompositeCommand

↓

CreateProjectCommand

CreatePageCommand

CreateBubbleCommand
```

Composite commands behave as a single operation.

---

# Macro Commands

Users may record command sequences.

Example

```
Resize Bubble

↓

Move Bubble

↓

Update Translation

↓

Macro
```

Macros replay commands without special-case logic.

---

# History

History stores executed commands.

Example

```text
History

↓

CreatePageCommand

↓

MoveBubbleCommand

↓

TranslateBubbleCommand
```

History is independent of the user interface.

---

# AI Integration

AI does not modify documents directly.

AI generates commands.

Example

```
TranslateBubbleCommand

↓

Handler

↓

Document
```

This guarantees that AI actions support:

- Undo
- History
- Events
- Transactions

---

# Plugin Integration

Plugins execute commands through the public API.

Plugins must never modify document objects directly.

---

# Event Integration

Successful command execution may publish domain events.

Examples

```
PageCreated

BubbleMoved

TranslationUpdated
```

Commands do not publish events directly.

Handlers coordinate event publication.

---

# Validation

Validation occurs before document mutation.

Validation failures prevent command execution.

Invalid commands never modify the document.

---

# Error Handling

Command execution returns structured results.

Example

```
Success

Failure

ValidationError

ConflictError
```

Exceptions are reserved for unexpected failures.

---

# Public API

Consumers execute commands through the Command Bus.

Good

```ts
await commandBus.execute(
    new CreatePageCommand(...)
);
```

Bad

```ts
project.pages.push(page);
```

Bad

```ts
project.addPage(page);
```

from outside the Command Framework.

---

# Package Responsibilities

atlas-command

Responsible for:

- Command definitions
- Command Bus
- Command Handlers
- Transactions
- Undo
- Redo
- Macro
- History

atlas-document

Responsible only for document state.

atlas-events

Responsible only for event dispatching.

atlas-core

Responsible only for business services.

---

# Rationale

The Command Pattern provides:

- Undo / Redo
- History
- Transactions
- Automation
- AI Integration
- Plugin Support
- Audit Trail

with a single execution model.

---

# Consequences

## Positive

- Consistent document mutations.
- Complete history.
- Predictable undo/redo.
- Easier testing.
- Better plugin architecture.
- Better AI integration.

## Negative

- More classes.
- Additional abstraction.
- Slightly more boilerplate.

---

# Alternatives Considered

## Direct Document Mutation

Rejected.

Reason:

No centralized execution model.

Undo and history become difficult.

---

## Service-Based Editing

Rejected.

Reason:

Services become large and difficult to compose.

---

## Event Sourcing

Rejected.

Reason:

Adds unnecessary complexity for the initial architecture.

May be reconsidered in a future ADR.

---

# Compliance

Every document modification must satisfy:

- [ ] Represented by a Command.
- [ ] Executed through the Command Bus.
- [ ] Handled by a Command Handler.
- [ ] Supports validation.
- [ ] Produces structured results.
- [ ] Can participate in transactions.
- [ ] Can participate in history.
- [ ] Does not modify documents directly.

Any exception requires a new ADR.
