# TASK-0022 — Define Package Export Map

## Summary

Define the official Package Export Map for the Atlas Studio Runtime Framework.

The Export Map controls the visibility of every module exposed by the package and serves as the navigation architecture for framework consumers.

It defines which entry points are public, which are feature-specific, and which remain internal.

---

# Goal

Provide a predictable, stable, and tree-shakeable package structure while preventing accidental exposure of internal implementation details.

---

# Background

As Atlas Studio grows, package structure becomes increasingly important.

A well-defined export map provides:

- Stable imports
- IDE auto-completion
- Tree shaking
- Internal encapsulation
- Backward compatibility

Instead of exposing the entire source tree, Atlas Studio exposes only carefully selected entry points.

---

# Scope

## Included

- package.json exports
- Root entry point
- Feature entry points
- Type exports
- ESM/CJS compatibility
- TypeScript declarations
- Export documentation

---

## Excluded

- Runtime implementation
- Build configuration
- Bundler optimization
- Publishing pipeline

# Export Categories

## Root Export

```ts
@atlas/core
```

Contains:

- Application
- ApplicationBuilder
- Configuration
- Environment
- Bootstrap

---

## Feature Exports

```text
@atlas/core/application

@atlas/core/runtime

@atlas/core/module

@atlas/core/provider

@atlas/core/resource

@atlas/core/configuration

@atlas/core/environment

@atlas/core/bootstrap

@atlas/core/registry
```

Each feature exports only stable contracts.

---

## Internal Exports

The following must never be exported:

```text
DefaultRuntimeKernel

RuntimeCoordinator

RuntimeStateMachine

DefaultBootstrap

BootstrapPipeline

DefaultModuleRegistry

DefaultServiceRegistry

ResolutionEngine

DependencyValidator

RuntimeDiagnostics

BuilderValidator
```

Internal implementation remains private.

---

# Export Rules

Root export should expose only:

- Frequently used APIs
- Stable interfaces
- Public builders
- Public factories
- Public enums

Feature exports should expose:

- Feature-specific contracts
- Public utility types
- Feature builders

Internal implementation must never be accessible through imports.

# TypeScript Support

Each export must provide:

```text
JavaScript module

Type declarations

Source maps
```

Consumers should never need deep imports.

---

# Import Examples

Preferred:

```ts
import { Application } from '@atlas/core';

import { Module } from '@atlas/core/module';
```

Forbidden:

```ts
import { DefaultRuntimeKernel } from '@atlas/core/src/runtime/DefaultRuntimeKernel';
```

---

# Design Principles

The Export Map should be:

- Stable
- Minimal
- Predictable
- Tree-shakeable
- IDE friendly

Deep imports should never be required.

---

# Technical Requirements

The export map must:

- Support ESM.
- Support CommonJS where required.
- Preserve type declarations.
- Prevent accidental implementation leakage.
- Preserve backwards compatibility.

---

# Future Extension Points

Future exports may include:

```text
@atlas/core/testing

@atlas/core/devtools

@atlas/core/diagnostics

@atlas/core/experimental
```

These entry points should be introduced without breaking existing imports.
