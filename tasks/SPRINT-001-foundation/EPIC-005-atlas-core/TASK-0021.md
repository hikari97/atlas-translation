# TASK-0021 — Define Public API Surface

## Summary

Define the official public API surface for the Atlas Studio Runtime Framework.

The Public API Surface specifies every symbol that is intentionally exposed to framework consumers.

Everything not included in the Public API Surface is considered internal implementation detail.

---

# Goal

Provide a stable, minimal, and versionable public API that remains independent from internal runtime implementation.

---

# Background

A runtime framework naturally grows over time.

Without a clearly defined public surface, internal implementation classes can accidentally become part of the supported API.

The Public API Surface acts as the contract between Atlas Studio and application developers.

---

# Scope

## Included

- Public exports
- Stable namespaces
- Entry points
- Public type aliases
- Public interfaces
- Public enums
- API organization
- API stability policy

---

## Excluded

- Runtime implementation
- Internal classes
- Diagnostics internals
- Registry internals
- Bootstrap internals
- Testing utilities

# Public Namespaces

The following namespaces are officially supported:

```text
application/

runtime/

configuration/

environment/

resource/

module/

registry/

provider/

bootstrap/
```

Each namespace exports only public contracts.

---

# Root Exports

The root package should export only the most frequently used symbols.

Example:

```ts
Application;

ApplicationBuilder;

Module;

Bootstrap;

Configuration;

Environment;
```

Advanced runtime components remain available through subpath exports.

---

# Public API Rules

Public API may expose:

- Interfaces
- Type aliases
- Enums
- Public builders
- Public factory functions
- Stable façade classes

Public API must never expose:

- Default implementations
- Internal coordinators
- Internal diagnostics
- Validators
- Runtime engines
- Internal state machines

---

# Versioning Policy

Public API follows Semantic Versioning.

Patch releases:

- Bug fixes only.

Minor releases:

- New public APIs.
- Backward compatible additions.

Major releases:

- Breaking API changes.

# Design Principles

The Public API should be:

- Minimal
- Stable
- Discoverable
- Strongly typed
- Framework independent

Developers should rarely need subpath imports.

---

# Technical Requirements

The implementation must:

- Export only stable symbols.
- Hide implementation details.
- Avoid duplicate exports.
- Avoid circular exports.
- Preserve tree shaking.
- Preserve type safety.

---

# Internal Visibility

The following must remain internal:

```text
DefaultRuntimeKernel

DefaultServiceRegistry

DefaultModuleRegistry

DefaultBootstrap

RuntimeCoordinator

RuntimeStateMachine

ResolutionEngine

DependencyValidator

BootstrapPipeline

ResourceDiagnostics
```

These classes are not part of the supported public API.

---

# Public Entry Point

Example usage:

```ts
import { Application, ApplicationBuilder } from '@atlas/core';
```

Advanced usage:

```ts
import { Module } from '@atlas/core/module';

import { Configuration } from '@atlas/core/configuration';
```
