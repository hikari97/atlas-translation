# TASK-0016 — Implement Application

## Summary

Implement the default `Application` for the Atlas Studio Runtime Framework.

The `Application` is the primary public API exposed to developers.

It acts as a lightweight façade over the Runtime Kernel and provides a stable, framework-independent interface for interacting with the runtime after bootstrap has completed.

---

# Goal

Provide the default implementation of the `Application` contract while maintaining a minimal public surface and delegating all runtime responsibilities to the Runtime Kernel.

---

# Background

The Runtime Kernel is an internal implementation detail.

Framework consumers should never interact directly with the Runtime Kernel.

Instead, they work exclusively with the `Application` abstraction.

The Application exposes runtime state, lifecycle operations, and runtime capabilities through a simplified API.

# Responsibilities

## DefaultApplication

Responsible for:

- Providing the public runtime API.
- Delegating runtime operations.
- Exposing application state.
- Exposing runtime capabilities.

Not responsible for:

- Bootstrapping.
- Service registration.
- Dependency resolution.
- Module loading.
- Business logic.

---

## DefaultApplicationContext

Responsible for:

- Holding application metadata.
- Providing access to RuntimeContext.
- Exposing runtime identity.
- Providing immutable runtime information.

---

## ApplicationFacade

Responsible for:

- Forwarding requests to RuntimeKernel.
- Hiding runtime implementation details.
- Maintaining API stability.

---

## ApplicationDiagnostics

Responsible for:

- Reporting application status.
- Reporting runtime information.
- Reporting diagnostics.

---

## ApplicationStateMachine

Responsible for:

- Tracking application state.
- Validating state transitions.
- Synchronizing state with RuntimeKernel.

# Public API

The Application should expose:

```ts
start();

stop();

dispose();

state();

context();

capabilities();

services();

modules();
```

The API should remain minimal and implementation-agnostic.

---

# Design Principles

The Application should be:

- Minimal
- Immutable after construction
- Strongly typed
- Framework independent
- Stable across runtime implementations

The Application must never expose internal RuntimeKernel implementation details.

---

# Technical Requirements

The implementation must:

- Follow the `Application` contract.
- Delegate all runtime operations.
- Preserve public API stability.
- Avoid circular dependencies.
- Remain platform independent.
