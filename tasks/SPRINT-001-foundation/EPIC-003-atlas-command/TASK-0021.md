# TASK-0021 — Implement Composite Commands

## Summary

Implement Composite Commands that allow multiple commands to be grouped and executed as a single logical command.

A Composite Command behaves like any other command while internally coordinating the execution of child commands.

Composite Commands become the foundation for:

- Macro Commands
- Batch Operations
- Import Workflows
- Multi-object Editing

---

# Goal

Provide a reusable command composition mechanism without introducing new execution semantics.

Composite Commands must integrate seamlessly with:

- Command Bus
- Pipeline
- Transactions
- History
- Undo / Redo

---

# Background

Many application features require executing several commands as one logical operation.

Examples include:

- Duplicate Page
- Import Project
- Batch Rename
- Multi-object Move
- Plugin Automation

Rather than exposing multiple commands to the caller, Atlas Studio encapsulates them inside a Composite Command.

# Responsibilities

Composite Commands are responsible for:

- Grouping multiple commands.
- Preserving execution order.
- Exposing a single command abstraction.
- Providing execution metadata.

Composite Commands are NOT responsible for:

- Executing commands.
- Transactions.
- Validation.
- Handler resolution.
- Event publication.

Execution remains the responsibility of the Command Framework.

---

# Execution Model

```text
CompositeCommand

↓

Command A

↓

Command B

↓

Command C
```

The Composite Command represents one logical operation while internally containing multiple child commands.

# Design Principles

Composite Commands should be:

- Immutable after creation.
- Strongly typed.
- Deterministic.
- Framework independent.
- Composable.

Composite Commands may contain other Composite Commands.

---

# Technical Requirements

The implementation must:

- Depend only on:

```

@atlas/document
```

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Preserve insertion order.
