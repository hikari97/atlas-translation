# TASK-0019 — Implement Redo Stack

## Summary

Implement the Redo Stack responsible for managing redo history.

The Redo Stack maintains references to History Entries that become eligible for redo after a successful undo operation.

It stores references only.

It never stores command instances.

It never executes redo operations.

---

# Goal

Provide a lightweight mechanism for tracking redo history while remaining independent from command execution.

The Redo Stack completes the history subsystem together with the History Store and Undo Stack.

---

# Background

Redo history should exist only after an undo operation.

When a new command executes successfully:

- Undo Stack receives a new HistoryEntry reference.
- Redo Stack is cleared.

When Undo succeeds:

- The corresponding HistoryEntry reference moves from Undo Stack to Redo Stack.

When Redo succeeds:

- The reference moves back from Redo Stack to Undo Stack.

This behavior matches common desktop editors.

# Responsibilities

The Redo Stack is responsible for:

- Pushing HistoryEntry references.
- Popping HistoryEntry references.
- Peeking the next redo entry.
- Clearing the stack.
- Reporting stack size.

The Redo Stack is NOT responsible for:

- Executing redo.
- Executing commands.
- Replaying history.
- Mutating documents.
- Transactions.
- Validation.

---

# Stack Model

The Redo Stack stores references only.

```text
RedoStack

↓

HistoryEntryId

↓

HistoryStore
```

No command instances are stored.

---

# Lifecycle

Normal execution

```text
Execute Command

History Store

```

```text

↓

Redo Stack
```

Redo

```text

↓
```
