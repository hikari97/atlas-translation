# TASK-0007 — Implement Command Registry

## Summary

Implement the Command Registry responsible for registering and discovering command types.

The registry stores command definitions and provides lookup capabilities.

It does **not** execute commands or create command instances.

---

# Goal

Create a centralized registry that enables the Command Framework to discover supported command types.

The registry becomes the foundation for:

- Command Resolver
- Command Dispatcher
- Plugin registration
- Future command discovery

---

# Background

The Command Bus should not know every available command.

Instead, commands register themselves (or are registered during application startup).

The registry provides a single source of truth for available command types.

# Responsibilities

The Command Registry is responsible for:

- Registering command definitions.
- Preventing duplicate registrations.
- Looking up command definitions.
- Removing registrations.
- Enumerating registered commands.

The registry is **not** responsible for:

- Creating command instances.
- Executing commands.
- Resolving handlers.
- Running middleware.
- Validation.
- Transactions.

---

# Public API

Expected exports

---

# Design Principles

The registry should be:

- Thread-safe (where applicable)
- Generic
- Lightweight
- Fast lookup
- Framework independent

The registry should not contain business logic.

---

# Technical Requirements

The implementation must:

- Depend only on:

```
@atlas/types
@atlas/document
```

- Use TypeScript strict mode.
- Use an internal `Map` for storage.
- Prevent duplicate command registrations.
- Avoid circular dependencies.
