# TASK-0008 — Implement Handler Registry

## Summary

Implement the Handler Registry responsible for maintaining the mapping between command types and handler types.

The registry enables the Command Framework to discover which handler is responsible for executing a specific command.

The registry does not instantiate handlers or execute commands.

---

# Goal

Create a centralized registry that maps commands to their corresponding handlers.

The registry becomes the foundation for:

- Command Resolver
- Command Dispatcher
- Dependency Injection
- Plugin Registration

---

# Background

According to ADR-0004, commands and handlers are separate concepts.

A command represents an intention.

A handler contains the execution logic.

The framework must be able to discover the appropriate handler without hardcoded dependencies.

# Responsibilities

The Handler Registry is responsible for:

- Registering handler mappings.
- Looking up handlers for commands.
- Preventing duplicate registrations.
- Removing registrations.
- Enumerating registered handlers.

The registry is **not** responsible for:

- Creating handler instances.
- Executing handlers.
- Dispatching commands.
- Validation.
- Middleware.
- Transactions.

# Design Principles

The Handler Registry should be:

- Generic
- Strongly typed
- Lightweight
- Fast lookup
- Framework independent

The registry should not contain business logic.

---

# Registration Model

The registry stores mappings between:

```text
Command Type
        │
        ▼
Handler Type
```

Example

```text
CreateProjectCommand

↓

CreateProjectHandler
```

The registry must not store handler instances.

---

# Technical Requirements

The implementation must:

- Depend only on:

```
@atlas/types
@atlas/document
```

- Use TypeScript strict mode.
- Use an internal `Map`.
- Prevent duplicate registrations.
- Avoid circular dependencies.
- Remain implementation-independent.
