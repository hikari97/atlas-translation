# TASK-0023 — Build Public API

## Summary

Design and implement the official public API for the `atlas-command` package.

The public API defines the only supported entry points that downstream packages, applications, and plugins may consume.

Internal implementation details must remain private.

---

# Goal

Provide a stable, minimal, and versioned public API.

The API should expose only framework contracts and supported implementations while hiding internal architecture.

---

# Background

As the package grows, internal folder structures will evolve.

Consumers should never depend on internal file locations.

Instead, all supported exports must be centralized through a single package entry point.

This allows internal refactoring without introducing breaking changes.

---

# Scope

## Included

- Root package exports
- Public barrel files
- Export organization
- API review
- Removal of accidental exports
- Public API documentation

---

## Excluded

- New features
- Business logic
- Internal refactoring
- Testing
- Performance improvements

# Responsibilities

The Public API is responsible for:

- Exporting supported contracts.
- Exporting supported implementations.
- Hiding internal utilities.
- Providing a stable package surface.

The Public API is NOT responsible for:

- Internal module organization.
- Dependency Injection.
- Runtime behavior.
- Testing.

---

# Public Export Policy

The package should export only supported symbols.

Examples include:

- Command
- CommandHandler
- CommandBus
- CommandPipeline
- PipelineBehavior
- Transaction
- HistoryStore
- UndoStack
- RedoStack
- CompositeCommand
- MacroCommand

Internal utilities such as:

- Default registry helpers
- Internal factories
- Private helpers
- Internal errors

must remain private unless explicitly documented.

---

# Package Entry Point

The package entry point should expose a curated API.

Consumers should write:

```ts
import {
  Command,
  CommandBus,
  PipelineBehavior,
  HistoryStore,
} from '@atlas/command';
```

Consumers should never import from:

```text
src/internal
src/private
src/utils
```

---

# Technical Requirements

The implementation must:

- Export only supported symbols.
- Avoid duplicate exports.
- Avoid circular exports.
- Preserve TypeScript declaration generation.
- Remain tree-shakable.
- Be backward compatible with previous public contracts.
