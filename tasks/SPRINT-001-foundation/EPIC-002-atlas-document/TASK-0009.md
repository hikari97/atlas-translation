# TASK-0009 — Implement Document Collections

## Summary

Implement reusable collection classes for the Atlas Studio document model.

Document Collections provide a consistent API for managing child documents while encapsulating mutable state and preserving document integrity.

Collections are shared across Workspace, Project, Page, Layer, Bubble, and future document models.

---

# Goal

Create a generic collection framework that standardizes how document objects are stored, queried, iterated, and modified.

The collection layer should become the single implementation used throughout atlas-document.

---

# Background

Nearly every document owns child documents.

Examples include:

- Workspace → Projects
- Project → Pages
- Page → Layers
- Layer → Bubbles

Without a shared collection implementation each document would duplicate collection logic.

# Responsibilities

DocumentCollection is responsible for:

- Storing child documents.
- Registering documents.
- Removing documents.
- Updating documents.
- Fast lookup by identifier.
- Iteration.
- Read-only enumeration.
- Collection statistics.

DocumentCollection is **not** responsible for:

- Validation.
- Persistence.
- Rendering.
- Business rules.
- Events outside the collection boundary.

---

# Public API

Examples include:

```ts
collection.size;

collection.isEmpty();

collection.has(id);

collection.get(id);

collection.add(document);

collection.remove(id);

collection.clear();

collection.values();

collection.keys();

collection.toArray();

collection.forEach();

collection.map();

collection.filter();

collection.find();
```

The implementation should provide a familiar, collection-oriented API while keeping internal storage encapsulated.

---

# Technical Requirements

The implementation must:

- Be generic (`DocumentCollection<T>`).
- Encapsulate mutable state.
- Preserve insertion order.
- Support efficient lookup by identifier.
- Expose read-only iteration.
- Use strict TypeScript.
- Depend only on `@atlas/types`.
