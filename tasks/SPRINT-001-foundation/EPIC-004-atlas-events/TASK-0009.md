# TASK-0009 — Implement Subscription Matcher

## Summary

Implement the Subscription Matcher responsible for selecting candidate subscriptions for published events.

The matcher evaluates registered subscriptions and determines which subscribers should receive a specific event.

The matcher never executes subscribers.

---

# Goal

Provide a reusable matching engine that converts a published event into an ordered collection of matching subscriptions.

The matcher should remain independent from the underlying registry implementation.

---

# Background

Publishing an event involves two independent steps:

1. Discover candidate subscriptions.
2. Execute matching subscribers.

The Subscription Matcher performs only the first step.

Subscriber execution is delegated to the Event Pipeline.

# Responsibilities

The Subscription Matcher is responsible for:

- Querying candidate subscriptions.
- Applying matching rules.
- Filtering disabled subscriptions.
- Ordering matching subscriptions.
- Producing matching results.

The Subscription Matcher is NOT responsible for:

- Registering subscriptions.
- Executing subscribers.
- Event publication.
- Pipeline execution.
- Replay.
- Retry.

# Design Principles

The Subscription Matcher should be:

- Stateless
- Deterministic
- Replaceable
- Framework independent
- Easy to test

The matcher should not depend directly on the Subscription Registry.

---

# Technical Requirements

The implementation must:

- Depend only on:

```
@atlas/types

@atlas/document

@atlas/command
```

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Support future wildcard subscriptions.
- Support future metadata matching.

---

# Matching Policies

The matcher should support:

- Exact event type
- Namespace matching
- Wildcard matching (future)
- Metadata filtering (future)
- Priority ordering
