# TASK-0005 — Define Event Subscriber Contracts

## Summary

Define the subscriber contracts used by the Atlas Studio Event Framework.

Subscribers consume published events through Subscription objects.

A published event may be delivered to zero or more subscribers.

This task defines contracts only.

No runtime implementation should be introduced.

---

# Goal

Provide strongly typed subscriber abstractions that enable a flexible Publish–Subscribe architecture.

These contracts become the public extension point for packages and plugins.

---

# Background

Unlike Commands, Events support multiple consumers.

Subscribers should never register directly with the Event Bus.

Instead, the Event Bus manages Subscription objects that associate one Event type with one Subscriber.

This indirection enables filtering, priorities, conditional delivery, replay, and future extensibility.

# Responsibilities

The Event Subscriber is responsible for:

- Receiving published events.
- Processing events.
- Returning subscriber execution results.

The Subscriber is NOT responsible for:

- Publishing events.
- Registering subscriptions.
- Event routing.
- Retry.
- Replay.
- Event persistence.

---

# Subscription Model

Each Subscription should define:

- Subscriber
- Event type
- Priority
- Optional filter
- Enabled/disabled state
- Metadata

Subscriptions are immutable after registration.

# Design Principles

Subscribers should be:

- Strongly typed
- Replaceable
- Testable
