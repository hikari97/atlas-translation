# TASK-0020 — Implement Event Journal Behavior

## Summary

Implement the Event Journal Behavior responsible for recording successfully published events into the Event Journal.

The behavior participates in the Publication Behavior Chain.

It records publication information without affecting event delivery.

---

# Goal

Provide a reusable publication behavior that transparently records published events through the Event Journal Writer.

Recording should be completely independent from business logic.

---

# Background

Every successfully published event may optionally be recorded.

Recorded events provide support for:

- Diagnostics
- Replay
- Audit
- Metrics
- Debugging

Recording should occur as part of the publication pipeline.

---

- EventJournalBehavior interface

# Responsibilities

## EventJournalBehavior

Responsible for:

- Recording published events.
- Building journal entries.
- Invoking EventJournalWriter.
- Enriching journal metadata.
- Continuing the Publication Behavior Chain.

Not responsible for:

- Publishing events.
- Subscriber execution.
- Replay.
- Business logic.
- Journal persistence implementation.

---

# Execution Flow

```text
Publication Behavior Chain

↓

Event Journal Behavior

↓

Journal Entry

↓

EventJournalWriter

↓

Event Journal

↓

Next Publication Behavior
```

Recording must not interrupt publication unless explicitly configured.

# Design Principles

The Event Journal Behavior should be:

- Stateless
- Replaceable
- Deterministic
- Framework independent
- Transparent to publishers

Journal recording should not modify the published event.

---

# Technical Requirements

The implementation must:

Depend only on:

```
@atlas/types

@atlas/document

@atlas/command
```

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Use EventJournalWriter for all recording operations.
- Preserve publication order.

---

# Recording Policy

The behavior should support future recording policies:

- Record every event.
- Record selected event types.
- Disable recording.
- Custom recording filters.

Policies should be configurable without modifying the behavior implementation.
