# TASK-0018 — Implement Event Replay Queue

## Summary

Implement the Event Replay Queue responsible for storing replay requests within the Atlas Studio Event Framework.

The Replay Queue schedules replay operations but does not execute them.

Replay execution is delegated to the Replay Coordinator.

---

# Goal

Provide a reusable queue that manages replay requests while remaining independent from replay execution.

---

# Background

Replay allows previously published events to be published again.

Replay requests may originate from:

- Debugging
- Diagnostics
- Testing
- Event recovery
- Development tools

The Replay Queue only stores replay requests.

Execution belongs to the Replay Coordinator.

# Responsibilities

## ReplayQueue

Responsible for:

- Accepting replay requests.
- Ordering replay requests.
- Storing pending requests.
- Returning queued requests.
- Removing completed requests.

Not responsible for:

- Executing replay.
- Publishing events.
- Subscriber execution.
- Business logic.

---

## ReplayQueueRepository

Responsible for:

- Persisting replay queue entries.
- Enumerating queued entries.
- Removing entries.
- Supporting future persistent storage.

# Design Principles

The Replay Queue should be:

- FIFO by default
- Priority-aware
- Deterministic
- Replaceable
- Framework independent

Queue management and replay execution must remain separate.

---

# Technical Requirements

The implementation must:

Depend only on:

```
@atlas/types

@atlas/document

@atlas/command
```

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Support future persistent repositories.
- Support priority ordering.

---

# Supported Operations

The queue must support:

- Enqueue
- Dequeue
- Peek
- Remove
- Clear
- Count
- Enumerate

Future extensions may include:

- Delayed replay
- Scheduled replay
- Batch replay
- Distributed replay queue
