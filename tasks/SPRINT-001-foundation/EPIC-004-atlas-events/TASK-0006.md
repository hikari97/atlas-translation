# TASK-0006 — Define Event Bus Contracts

## Summary

Define the Event Bus contracts used by the Atlas Studio Event Framework.

The Event Bus is the primary public entry point for publishing events.

It is responsible for accepting published events and coordinating their delivery to matching subscriptions.

This task defines contracts only.

No runtime implementation should be introduced.

---

# Goal

Provide a stable and minimal Event Bus abstraction that supports synchronous and asynchronous event publication while hiding the internal event routing mechanism.

---

# Background

The Event Bus is the façade of the Atlas Studio Event Framework.

Publishers should never interact directly with:

- Subscribers
- Subscription Registry
- Pipeline
- Replay
- Internal infrastructure

Instead, publishers interact exclusively with the Event Bus.

# Responsibilities

The Event Bus is responsible for:

- Accepting events from publishers.
- Starting the publication pipeline.
- Returning EventPublicationResult.
- Supporting synchronous and asynchronous publication.
- Remaining independent of subscriber implementations.

The Event Bus is NOT responsible for:

- Resolving subscribers.
- Executing subscribers.
- Event persistence.
- Replay.
- Business logic.

---

# Publication Flow

```text
Publisher

↓

EventBus

↓

Internal Subscription Registry

↓

Pipeline

↓

Subscribers

↓

EventPublicationResult
```

The Subscription Registry and Pipeline are internal implementation details.

# Design Principles

The Event Bus should be:

- Minimal
- Strongly typed
- Framework independent
- Extensible
- Easy to test

Consumers should interact only with the Event Bus.

---

# Technical Requirements

The implementation must:

- Depend only on:

```
@atlas/types

@atlas/document

@atlas/command
```

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Support future event replay.
- Support synchronous and asynchronous publication.

---

# Publication Modes

The framework should support:

- Synchronous publication
- Asynchronous publication

The publication mode is selected through `PublishOptions`.

---

# Publication Policies

Possible publication policies include:

- Publish to all subscribers.
- Stop on first subscriber failure.
- Continue after subscriber failures.
- Collect all subscriber errors.

Policies define publication behavior only and do not implement subscriber logic.
