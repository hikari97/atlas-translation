# TASK-0021 — Implement Event Composition

## Summary

Implement the Event Composition framework for the Atlas Studio Event Framework.

Event Composition enables multiple events to be grouped and published as a single logical operation while preserving the identity of each individual event.

The framework supports composite publication strategies without modifying the Event Bus.

---

# Goal

Provide a reusable abstraction for composing multiple events into a single publication unit while maintaining compatibility with the existing Event Pipeline.

---

# Background

Some application workflows naturally produce multiple related events.

Examples include:

- Importing a document
- Duplicating multiple objects
- Batch editing
- Undo/Redo operations
- Plugin initialization

Publishing each event individually may increase coordination complexity.

Event Composition groups related events while preserving publication semantics.

# Responsibilities

## EventComposition

Responsible for:

- Representing a logical collection of events.
- Preserving event ordering.
- Coordinating publication strategies.
- Exposing composed events.

Not responsible for:

- Publishing events.
- Subscriber execution.
- Replay.
- Persistence.
- Business logic.

---

## CompositeEvent

Responsible for:

- Holding multiple related events.
- Preserving insertion order.
- Publishing as one logical operation.

---

## BatchEvent

Responsible for:

- Representing independent events processed together.
- Optimizing bulk publication.

---

## AggregateEvent

Responsible for:

- Representing a higher-level event derived from multiple lower-level events.
- Preserving relationships between child events.

---

## ChainedEvent

Responsible for:

- Representing events that must be published sequentially.
- Preserving dependency order.

---

# Composition Flow

```text
Application

↓

EventComposition

↓

↓

Publication Pipeline

↓

Subscribers
```

Each composed event still passes through the normal publication pipeline.

# Design Principles

Event Composition should be:

- Immutable
- Deterministic
- Replaceable
- Strongly typed
- Framework independent

Composition must never alter the semantics of individual events.

---

# Technical Requirements

The implementation must depend only on:

```
@atlas/types

@atlas/document

@atlas/command
```

Additional requirements:

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Preserve event ordering.
- Support future publication strategies.

---

# Supported Composition Strategies

The framework should support:

- Sequential publication
- Batch publication
- Composite publication
- Aggregate publication

Future strategies may include:

- Parallel publication
- Conditional publication
- Transactional publication
