# TASK-0019 — Implement Replay Coordinator

## Summary

Implement the Replay Coordinator responsible for orchestrating event replay within the Atlas Studio Event Framework.

The Replay Coordinator consumes replay requests from the Replay Queue, retrieves recorded events from the Event Journal, and republishes them through the Event Bus.

The Replay Coordinator coordinates replay execution but does not implement replay scheduling or event persistence.

---

# Goal

Provide a reusable orchestration component that executes replay requests while remaining independent from storage and scheduling concerns.

---

# Background

Replay allows previously published events to be republished.

Typical use cases include:

- Debugging
- Diagnostics
- Recovery
- Testing
- Development tools

Replay must reuse the normal Event Bus so that replayed events traverse the same publication pipeline as live events.

# Responsibilities

## ReplayCoordinator

Responsible for:

- Consuming replay requests.
- Reading recorded events.
- Republishing events through the Event Bus.
- Aggregating replay statistics.
- Producing replay execution results.

Not responsible for:

- Scheduling replay.
- Recording events.
- Subscriber execution.
- Event matching.
- Business logic.

# Design Principles

The Replay Coordinator should be:

- Stateless
- Deterministic
- Replaceable
- Framework independent
- Easy to test

Replay execution should be indistinguishable from live publication except for replay metadata.

---

# Technical Requirements

The implementation must:

Depend only on:

```
@atlas/types

@atlas/document

@atlas/command
```

- Use TypeScript strict mode.
- Avoid runtime dependencies.
- Avoid circular dependencies.
- Reuse the Event Bus for publication.
- Preserve event ordering during replay.
