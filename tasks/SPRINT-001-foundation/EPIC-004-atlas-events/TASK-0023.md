# TASK-0023 — Build Public API

## Summary

Build the public API for the `atlas-events` package.

This task consolidates all stable public contracts and implementations into a clean, well-organized export surface.

Only stable APIs should be exported.

Internal framework infrastructure must remain private.

---

# Goal

Provide a minimal, consistent, and versionable public API that serves as the single entry point for consumers of the Atlas Studio Event Framework.

---

# Background

Throughout EPIC-004 the package introduced numerous contracts and implementations.

Consumers should not import modules from internal folders.

Instead, all public functionality is exposed through a single package entry point.

This task freezes the public API for downstream packages.

---

# Scope

## Included

- Public barrel exports
- Stable API surface
- Export organization
- Public type exports
- Package entry point
- Export validation

---

## Excluded

- Internal framework classes
- Internal coordinators
- Internal repositories
- Internal execution contexts
- Internal diagnostics

# Responsibilities

The Public API is responsible for:

- Exposing stable interfaces.
- Exposing stable implementations.
- Providing a single import location.
- Hiding internal implementation details.

The Public API is NOT responsible for:

- Runtime behavior.
- Business logic.
- Dependency injection.
- Internal orchestration.

---

# Public Modules

The package should expose:

```text
Event

EventContext

EventPublicationResult

EventBus

EventPublisher

Subscription

EventSubscriber

SubscriptionManager

EventPipeline

PublicationBehavior

SubscriberBehavior

EventJournal

ReplayQueue

ReplayCoordinator

EventFilter

EventComposition
```

Only stable APIs should be exported.

---

# Internal Modules

The following remain internal:

```text
DefaultEventPublicationCoordinator

DefaultSubscriptionIndex

SubscriptionMatcher

BehaviorOrdering

PipelineExecutionContext

JournalRepository

ReplayQueueRepository

ExecutionResult

Internal Errors
```

Internal APIs must never be imported directly by consumers.

---

# Design Principles

The public API should be:

- Minimal
- Stable
- Predictable
- Well organized
- Versionable

Internal implementation details should remain hidden.

---

# Technical Requirements

The implementation must:

- Use barrel exports.
- Avoid duplicate exports.
- Avoid circular exports.
- Preserve tree-shaking.
- Preserve ESM compatibility.
- Preserve CommonJS compatibility.

---

# Export Rules

Consumers should write:

```ts
import {
  EventBus,
  Event,
  EventSubscriber,
  EventJournal,
  ReplayCoordinator,
} from '@atlas/events';
```

Consumers should never import from:

```text
src/internal

src/registry

src/pipeline/internal

src/behaviors/internal
```
