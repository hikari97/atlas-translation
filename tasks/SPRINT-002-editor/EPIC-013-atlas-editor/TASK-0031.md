# TASK-0031 — Implement Integration Context

## Summary

Implement the `IntegrationContext`, the root abstraction responsible for exposing the shared integration environment used by every Atlas Studio subsystem.

The Integration Context serves as the composition boundary for cross-package communication. It provides immutable access to shared registries, lifecycle coordinators, capability providers, dependency graphs, extension hosts, event dispatchers, and future integration services while remaining completely independent from rendering engines, browser APIs, operating systems, storage providers, and application frameworks.

The Integration Context is responsible only for exposing integration services. It does not execute commands, render graphics, instantiate services, or implement business logic.

---

# Goal

Provide a reusable, immutable, deterministic, and framework-independent Integration Context capable of exposing every integration service through a stable public API.

---

# Background

Atlas Studio consists of many independent packages.

Examples include:

- atlas-editor
- atlas-document
- atlas-renderer
- atlas-history
- atlas-selection
- atlas-input
- atlas-command
- atlas-plugin
- atlas-translation
- atlas-export

These packages should communicate through contracts rather than direct references.

Atlas Studio should expose one unified Integration Context.

# Responsibilities

The Integration Context is responsible for:

- Exposing integration services.
- Supporting dependency injection.
- Supporting runtime inspection.
- Providing immutable service references.
- Supporting future integration modules.
- Preserving deterministic dependency composition.

The Integration Context is not responsible for:

- Executing services.
- Creating services.
- Rendering.
- Business logic.
