# TASK-0035 — Implement Extension Host

## Summary

Implement the `ExtensionHost`, the runtime abstraction responsible for discovering, loading, registering, isolating, activating, deactivating, and unloading every Atlas Studio extension.

The Extension Host serves as the runtime boundary between Atlas Studio core packages and third-party extensions. It provides deterministic extension lifecycle management while remaining completely independent from rendering engines, browser APIs, operating systems, module loaders, and application frameworks.

The Extension Host is responsible only for extension orchestration. It does not execute extension business logic, render graphics, or implement dependency injection.

---

# Goal

Provide a reusable, deterministic, strongly typed, framework-independent Extension Host capable of managing every Atlas Studio extension through a stable public API.

---

# Background

Atlas Studio is designed as a highly extensible editor.

Future extensions may provide:

- OCR Engines
- Translation Providers
- AI Models
- Renderers
- Importers
- Exporters
- Themes
- Custom Tools
- Commands
- Panels
- Workflows
- Automation

Extensions must never communicate directly with editor internals.

All communication must occur through stable contracts.

# Responsibilities

The Extension Host is responsible for:

- Registering extensions.
- Activating extensions.
- Deactivating extensions.
- Managing extension lifecycle.
- Providing extension context.
- Supporting runtime inspection.
- Preserving deterministic execution.

The Extension Host is **not** responsible for:

- Installing extensions.
- Downloading packages.
- Rendering.
- Business logic.
- Dependency injection.

# Public API

```ts
interface ExtensionHost {
  register(extension: Extension): void;

  unregister(id: string): void;

  activate(id: string): Promise<void>;

  deactivate(id: string): Promise<void>;

  get(id: string): Extension | undefined;

  list(): readonly Extension[];
}
```
