# TASK-0009 — Implement Workspace Context

## Summary

Implement the `WorkspaceContext`, the runtime dependency composition abstraction responsible for exposing every service owned by a Workspace.

The Workspace Context acts as the dependency boundary between the Editor runtime and Workspace runtime. It provides immutable access to all workspace-scoped services including documents, renderer, history, selection, translation, input, commands, plugins, tools, sessions, and future workspace modules while remaining completely framework independent.

The Workspace Context is responsible only for exposing runtime dependencies. It does not create services, own service lifecycles, execute business logic, or render interfaces.

---

# Goal

Provide a reusable, immutable, deterministic, and strongly typed Workspace Context capable of exposing every workspace service through a stable public API.

---

# Background

Every Workspace owns its own runtime.

Examples include:

- Document Manager
- Renderer Context
- History Context
- Selection Context
- Input Context
- Translation Context
- Tool Manager
- Plugin Context

Without a dedicated Workspace Context, runtime dependencies become tightly coupled and difficult to replace.

Atlas Studio should expose one immutable Workspace Context abstraction.

# Responsibilities

The Workspace Context is responsible for:

- Exposing workspace services.
- Supporting dependency injection.
- Preserving immutable runtime references.
- Supporting runtime inspection.
- Supporting future runtime extensions.
- Maintaining deterministic dependency resolution.

The Workspace Context is **not** responsible for:

- Creating services.
- Managing service lifecycles.
- Executing commands.
- Rendering.
- Implementing business logic.
