# TASK-0038 — Implement Runtime Profiler

## Summary

Implement the `RuntimeProfiler`, the abstraction responsible for profiling runtime execution across every Atlas Studio subsystem.

The Runtime Profiler captures execution timelines, nested operations, resource utilization, asynchronous execution spans, plugin execution, AI inference timing, rendering phases, and future runtime activities while remaining completely independent from rendering engines, browser APIs, operating systems, and application frameworks.

The Runtime Profiler is responsible only for collecting execution profiles. It does not optimize execution, modify runtime behavior, or implement business logic.

---

# Goal

Provide a reusable, deterministic, lightweight, strongly typed, and framework-independent Runtime Profiler capable of exposing execution profiles through a stable public API.

---

# Background

Atlas Studio performs many expensive runtime operations.

Examples include:

- OCR
- AI Translation
- Inpainting
- Rendering
- Export
- Autosave
- Plugin execution
- Event dispatching

Performance metrics alone cannot explain where execution time is spent.

A Runtime Profiler should expose execution spans and timing relationships.

# Responsibilities

The Runtime Profiler is responsible for:

- Creating execution spans.
- Recording nested operations.
- Recording asynchronous execution.
- Building execution timelines.
- Supporting runtime inspection.
- Preserving deterministic profiling.

The Runtime Profiler is **not** responsible for:

- Optimizing execution.
- Rendering charts.
- Uploading telemetry.
- Implementing business logic.

# Public API

```ts
interface RuntimeProfiler {
  beginSpan(name: RuntimeSpanToken): RuntimeSpan;

  profile<T>(name: RuntimeSpanToken, operation: () => Promise<T>): Promise<T>;

  currentSpan(): RuntimeSpan | null;

  timeline(): RuntimeTimeline;
}
```

# Profiling Rules

The implementation should support:

- Nested spans.
- Parallel spans.
- Asynchronous spans.
- Immutable timelines.
- Correlation IDs.
- Parent-child relationships.

Profiling must never modify runtime behavior.
