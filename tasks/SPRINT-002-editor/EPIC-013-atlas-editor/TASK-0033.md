# TASK-0033 — Implement Service Registry

## Summary

Implement the `ServiceRegistry`, the canonical abstraction responsible for registering, resolving, validating, and exposing runtime services used throughout Atlas Studio.

The Service Registry provides deterministic dependency resolution for every Atlas Studio package while remaining completely independent from dependency injection frameworks, rendering engines, browser APIs, operating systems, and application frameworks.

The Service Registry is responsible only for service registration and resolution. It does not instantiate services, manage business logic, or execute runtime behavior.

---

# Goal

Provide a reusable, deterministic, immutable, strongly typed, and framework-independent Service Registry capable of exposing runtime services through a stable public API.

---

# Background

Atlas Studio consists of many independent packages.

Examples:

- atlas-document
- atlas-renderer
- atlas-history
- atlas-selection
- atlas-input
- atlas-command
- atlas-plugin
- atlas-translation
- atlas-export

Every package should depend on service contracts rather than concrete implementations.

# Responsibilities

The Service Registry is responsible for:

- Registering services.
- Resolving services.
- Validating registrations.
- Tracking service metadata.
- Supporting plugin-defined services.
- Preserving deterministic lookup behavior.

The Service Registry is **not** responsible for:

- Instantiating services.
- Managing object lifetimes.
- Dependency injection.
- Rendering.
- Business logic.

# Public API

```ts
interface ServiceRegistry {
  register<T>(descriptor: ServiceDescriptor<T>): void;

  unregister(id: string): void;

  resolve<T>(id: string): T;

  tryResolve<T>(id: string): T | undefined;

  contains(id: string): boolean;

  list(): readonly ServiceDescriptor[];
}
```

---

# Registration Rules

The implementation should support:

- Unique service identifiers.
- Immutable descriptors.
- Contract-first registration.
- Runtime validation.
- Version compatibility.
- Plugin registrations.

Service lookup must remain deterministic.

---

# Service Lifetime

The implementation should expose lifetime metadata including:

- Singleton
- Scoped
- Transient
- External

The registry exposes metadata only.

It must never manage object lifetimes.

---

The Service Registry should be:

- Framework independent
- Strongly typed
- Immutable

- Deterministic
- Extensible

The registry is **not** a Dependency Injection Container.
