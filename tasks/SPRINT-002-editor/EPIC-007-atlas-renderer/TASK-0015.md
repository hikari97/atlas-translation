# TASK-0015 — Implement Patch Operations

## Summary

Implement the Patch Operations subsystem for Atlas Studio.

Patch Operations define the atomic modifications that can be applied to a rendered output. Rather than allowing the Patch Engine to perform arbitrary mutations, every change must be represented as a strongly typed patch operation.

This abstraction keeps the rendering backend independent from the diffing algorithm while enabling incremental rendering.

---

# Goal

Provide a standardized set of immutable patch operations that can be executed by any rendering backend.

---

# Background

The Diff Engine identifies structural changes.

The Patch Engine executes those changes.

Patch Operations define **what** changes need to occur.

By representing every mutation as an operation object, Atlas Renderer gains:

- Deterministic execution
- Replay capability
- Easier debugging
- Backend independence
- Future optimization opportunities

# Responsibilities

Patch Operations are responsible for:

- Representing atomic render updates.
- Defining supported mutation types.
- Providing immutable operation objects.
- Supporting validation before execution.
- Enabling backend-independent patch execution.

Patch Operations are **not** responsible for:

- Executing operations.
- Comparing trees.
- Rendering.
- Scheduling.
- Runtime management.

---

# Supported Operations

The following operations should be supported:

- CreateNode
- RemoveNode
- ReplaceNode
- MoveNode
- UpdateAttributes
- UpdateProperties
- InsertChild
- RemoveChild

Future operation types should remain backward compatible.

Example collection:

```ts
interface PatchOperationCollection {
  readonly operations: readonly PatchOperation[];
}
```

---

# Design Principles

Patch Operations should be:

- Immutable
- Serializable
- Strongly typed
- Backend independent
- Easy to validate

Operations should contain only data and never execution logic.

---

# Technical Requirements

The implementation must:

- Define reusable operation types.
- Support immutable collections.
- Validate required fields.
- Preserve execution ordering.
- Preserve public API stability.
