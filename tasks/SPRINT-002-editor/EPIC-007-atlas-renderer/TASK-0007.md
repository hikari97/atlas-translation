# TASK-0007 — Implement Render Node

## Summary

Implement the Render Node model for Atlas Studio.

Render Node represents a single element within the Render Tree. Every visual element produced by the renderer is represented by a Render Node.

Render Nodes form the fundamental building blocks of the rendering pipeline and are consumed by the Diff Engine, Patch Engine, and Layout System.

---

# Goal

Provide a strongly typed, immutable, and hierarchical node model that represents rendered elements independently from any rendering backend.

---

# Background

While the Render Tree manages the overall tree structure, individual nodes are responsible for representing renderable elements.

A Render Node should contain only rendering information and must never modify the original Atlas Document.

# Responsibilities

Render Node is responsible for:

- Representing one rendered element.
- Maintaining parent-child relationships.
- Storing rendering properties.
- Exposing immutable attributes.
- Supporting traversal.

Render Node is **not** responsible for:

- Rendering.
- Scheduling.
- Diff calculation.
- Patch generation.
- Layout computation.

---

# Node Structure

Example:

```text
RenderNode

├── id
├── type
├── attributes
├── properties
├── parent
└── children
```

Each Render Node belongs to exactly one Render Tree.

---

# Public API

Example:

```ts
interface RenderNode {
  readonly id: string;

  readonly type: RenderNodeType;

  readonly parent?: RenderNode;

  readonly children: readonly RenderNode[];

  readonly attributes: RenderAttributes;

  readonly properties: RenderProperties;
}
```

---

# Design Principles

Render Node should be:

- Immutable
- Strongly typed
- Lightweight
- Hierarchical
- Platform independent

Node instances should not contain rendering logic.

---

# Technical Requirements

The implementation must:

- Preserve parent-child consistency.
- Support immutable children collections.
- Prevent circular references.
- Support efficient traversal.
- Preserve public API stability.
