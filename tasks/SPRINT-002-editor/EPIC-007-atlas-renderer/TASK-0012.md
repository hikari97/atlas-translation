# TASK-0012 — Implement Render Session

## Summary

Implement the Render Session subsystem for Atlas Studio.

A Render Session represents a single rendering execution from start to completion. It stores transient rendering state, execution metadata, timing information, and intermediate artifacts produced during the rendering process.

The Render Session exists only for the lifetime of a render operation and must never become part of the persistent Render Tree.

---

# Goal

Provide an isolated execution context that encapsulates all temporary state required during a rendering operation.

---

# Background

Rendering consists of multiple processing stages.

Rather than storing temporary state inside the Render Engine or Render Pipeline, every rendering operation receives its own Render Session.

This approach enables:

- Concurrent rendering
- Incremental rendering
- Render cancellation
- Better diagnostics
- Performance measurement

# Responsibilities

Render Session is responsible for:

- Tracking render execution.
- Storing temporary rendering state.
- Recording execution timing.
- Collecting rendering statistics.
- Providing session metadata.

Render Session is **not** responsible for:

- Rendering.
- Scheduling.
- Tree construction.
- Diff calculation.
- Patch execution.

---

# Session Lifecycle

```text
Created

↓

Initialized

↓

Running

↓

Completed

↓

Disposed
```

A Render Session must never be reused after completion.

---

# Public API

Example:

```ts
interface RenderSession {
  readonly id: string;

  readonly state: RenderSessionState;

  readonly metadata: RenderSessionMetadata;

  readonly statistics: RenderSessionStatistics;

  start(): void;

  complete(): void;

  dispose(): void;
}
```

---

# Design Principles

Render Session should be:

- Ephemeral
- Isolated
- Strongly typed
- Immutable where appropriate
- Easy to inspect

Session state should exist only during a single render execution.

---

# Technical Requirements

The implementation must:

- Generate unique session identifiers.
- Record execution timestamps.
- Track lifecycle state.
- Collect rendering statistics.
- Preserve public API stability.
