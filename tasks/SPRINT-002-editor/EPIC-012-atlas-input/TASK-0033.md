# TASK-0033 — Implement Event Bus

## Summary

Implement the `EventBus`, the central publish–subscribe abstraction responsible for decoupled event communication throughout Atlas Studio.

The Event Bus provides a deterministic messaging infrastructure that enables independent packages—including editor, renderer, document, history, plugins, developer tools, and future modules—to exchange immutable events without requiring direct dependencies. It supports topic-based subscriptions, event broadcasting, scoped channels, event filtering, and extensible message routing while remaining completely independent from rendering engines, browser APIs, operating system messaging systems, application frameworks, and editor implementations.

The Event Bus is responsible only for event publication and subscription. It does not generate events, execute commands, render interfaces, modify application state, or implement business logic.

---

# Goal

Provide a reusable, deterministic, immutable, and framework-independent publish–subscribe abstraction capable of enabling loosely coupled communication across every Atlas Studio package through a stable public API.

---

# Background

As Atlas Studio grows, packages should communicate without creating direct dependencies.

Typical publishers include:

- Input Manager
- Shortcut Manager
- Renderer
- History
- Document
- Plugin Host
- Developer Tools

Typical subscribers include:

- Editor
- Canvas
- Selection
- Inspector
- Timeline
- Plugins
- Analytics
- Accessibility

Rather than tightly coupling producers and consumers, Atlas Studio should expose a unified Event Bus abstraction.

# Public API

Example:

```ts
interface EventBus {
  publish<T>(topic: string, event: T): void;

  subscribe<T>(topic: string, handler: EventHandler<T>): EventSubscription;

  unsubscribe(subscription: EventSubscription): void;

  clear(): void;
}
```

Example subscription:

```ts
interface EventSubscription {
  readonly id: string;

  readonly topic: string;

  readonly createdAt: number;
}
```
