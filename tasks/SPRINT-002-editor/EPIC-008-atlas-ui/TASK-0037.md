# TASK-0037 — Implement Panel Manager

## Summary

Implement the Panel Manager for Atlas UI.

The Panel Manager provides a unified mechanism for registering, organizing, activating, hiding, showing, docking, and restoring workspace panels. It integrates with the Workspace Layout, Docking System, Split View, Theme System, Accessibility Layer, and Layout Services while remaining framework-independent.

The Panel Manager is responsible only for coordinating panel presentation. Panel content, business logic, and workspace state remain outside this component.

---

# Goal

Provide a reusable, extensible, and accessible Panel Manager capable of orchestrating every workspace panel through a stable public API.

---

# Background

Modern IDEs and editors typically contain dozens of independent panels.

Examples include:

- Explorer
- Outline
- Search
- Terminal
- Console
- Inspector
- Extensions
- Plugin Panels

Instead of every subsystem managing its own panel lifecycle, Atlas UI should expose a centralized Panel Manager.

# Responsibilities

The Panel Manager is responsible for:

- Registering workspace panels.
- Managing panel visibility.
- Managing active panels.
- Organizing panel groups.
- Integrating with Theme System.
- Providing accessibility metadata.

The Panel Manager is **not** responsible for:

- Rendering panel contents.
- Docking implementation.
- Workspace persistence.
- Window management.
- Business logic.

# Public API

Example:

```ts
interface PanelManager extends UIComponent {
  readonly props: PanelManagerProps;

  register(panel: WorkspacePanel): void;

  unregister(panelId: string): void;

  activate(panelId: string): void;

  deactivate(panelId: string): void;
}
```

Example panel:

```ts
interface WorkspacePanel {
  readonly id: string;

  readonly title: string;

  readonly icon?: UIElement;

  readonly group: string;

  readonly provider: PanelProvider;
}
```

# Design Principles

The Panel Manager should be:

- Provider-driven
- Accessible
- Theme-aware
- Framework independent
- Strongly typed

Panels should be registered dynamically rather than hard-coded.

---

# Technical Requirements

The implementation must:

- Support dynamic registration.
- Support lazy loading.
- Preserve immutable panel definitions.
- Avoid hard-coded styling.
- Preserve API stability.

---

# Accessibility Requirements

The Panel Manager must:

- Support keyboard navigation.
- Preserve logical focus order.
- Support screen readers.
- Respect accessibility settings.
- Maintain panel activation semantics.
