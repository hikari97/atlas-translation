# TASK-0024 — Implement Drag Constraints

## Summary

Implement the `DragConstraints`, a reusable abstraction responsible for defining and validating the movement boundaries of an active drag session.

The Drag Constraints component allows drag operations to be limited by configurable rules such as axis locking, bounding regions, movement thresholds, snapping policies, and custom validation callbacks.

The Drag Constraints abstraction represents only constraint metadata and validation rules. It does not move objects, modify layouts, or perform rendering.

---

# Goal

Provide a reusable, extensible, and framework-independent drag constraint abstraction capable of validating drag movement through a stable public API.

---

# Background

Most drag operations cannot move freely.

Typical constraints include:

- Canvas boundaries
- Workspace boundaries
- Horizontal-only dragging
- Vertical-only dragging
- Grid snapping
- Parent container bounds
- Plugin-defined constraints

Instead of embedding constraint logic into every draggable component, Atlas Studio should expose a unified Drag Constraints abstraction.

# Responsibilities

The Drag Constraints component is responsible for:

- Defining drag boundaries.
- Validating proposed movement.
- Applying axis restrictions.
- Providing immutable constraint metadata.
- Supporting custom validation rules.

The Drag Constraints component is **not** responsible for:

- Moving dragged objects.
- Updating layouts.
- Rendering drag previews.
- Executing commands.
- Business logic.

# Public API

Example:

```ts
interface DragConstraints {
  readonly rules: readonly DragConstraintRule[];

  validate(position: DragPosition): DragConstraintResult;
}
```

Example rule:

```ts
interface DragConstraintRule {
  readonly id: string;

  readonly type: string;

  readonly enabled: boolean;
}
```

Example result:

```ts
interface DragConstraintResult {
  readonly accepted: boolean;

  readonly correctedPosition?: DragPosition;
}
```

# Design Principles

The Drag Constraints component should be:

- Framework independent
- Immutable
- Strongly typed
- Lightweight
- Extensible

Constraint evaluation should remain deterministic and independent from rendering or application logic.

# Technical Requirements

The implementation must:

- Support multiple active constraint rules.
- Preserve immutable validation results.

- Support custom constraint providers.
- Preserve API stability.
