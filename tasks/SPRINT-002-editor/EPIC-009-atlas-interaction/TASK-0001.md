# TASK-0001 — Implement Interaction Manager

## Summary

Implement the `InteractionManager`, the central coordinator responsible for managing interaction sessions throughout Atlas Studio.

The Interaction Manager provides a unified entry point for registering, activating, cancelling, and routing interaction sessions. It serves as the foundation for all interaction types, including pointer, mouse, keyboard, drag-and-drop, resize, and gesture interactions.

The manager coordinates interaction lifecycles but does not perform rendering, modify documents, or execute application logic.

---

# Goal

Provide a reusable, extensible, and framework-independent Interaction Manager capable of coordinating all interaction sessions through a stable public API.

---

# Background

Every user interaction inside Atlas Studio should pass through a common interaction layer.

Rather than allowing individual subsystems to independently manage interaction lifecycles, a centralized manager ensures consistent behavior, predictable state transitions, and future extensibility.

# Responsibilities

The Interaction Manager is responsible for:

- Registering interaction sessions.
- Managing interaction lifecycle.
- Activating sessions.
- Cancelling sessions.
- Tracking the active interaction.
- Routing interaction requests.

The Interaction Manager is **not** responsible for:

- Rendering.
- Editing documents.
- Executing commands.
- Managing selections.
- Workspace management.

# Public API

Example:

```ts
interface InteractionManager {
  register(session: InteractionSession): void;

  unregister(id: string): void;

  activate(id: string): void;

  cancel(id: string): void;

  active(): InteractionSession | undefined;
}
```

# Design Principles

The Interaction Manager should be:

- Framework independent
- Strongly typed
- Event driven
- Extensible
- Stateless whenever possible

Interaction state should remain inside individual interaction sessions.
