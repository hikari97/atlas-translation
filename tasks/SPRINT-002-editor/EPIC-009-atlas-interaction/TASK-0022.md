# TASK-0022 — Implement Drop Session

## Summary

Implement the `DropSession`, a reusable abstraction responsible for representing the completion of a drag-and-drop interaction within Atlas Studio.

A Drop Session encapsulates the lifecycle of a drop operation by identifying the source interaction, validating the drop target, and exposing immutable metadata describing the completed interaction.

The Drop Session is responsible only for representing drop interaction state. It does not modify documents, update layouts, execute commands, or render visual feedback.

---

# Goal

Provide a reusable, extensible, and framework-independent drop session abstraction capable of representing drop operations through a stable public API.

---

# Background

Drop operations finalize drag interactions.

Typical examples include:

- Dropping editor blocks
- Reordering panels
- Reordering tabs
- Moving tree nodes
- Moving selections
- Dropping floating windows
- Plugin-defined drop targets

Rather than allowing each subsystem to define its own drop lifecycle, Atlas Studio should expose a unified Drop Session abstraction.

# Responsibilities

The Drop Session is responsible for:

- Representing a completed drop interaction.
- Recording drag source information.
- Recording drop target information.
- Tracking drop lifecycle.
- Providing immutable drop metadata.

The Drop Session is **not** responsible for:

- Moving objects.
- Updating layouts.
- Executing commands.
- Rendering previews.
- Business logic.

# Public API

Example:

```ts
interface DropSession {
  readonly id: string;

  readonly dragSessionId: string;

  readonly target: DropTarget;

  readonly state: DropSessionState;

  readonly timestamp: number;
```

```ts
interface DropTarget {
```

The Drop Session should be:

- Framework independent
- Immutable
- Strongly typed
- Lightweight
