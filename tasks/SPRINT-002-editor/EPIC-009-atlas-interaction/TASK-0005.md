# TASK-0005 — Implement Interaction Registry

## Summary

Implement the `InteractionRegistry`, a centralized registry responsible for managing available interaction types within Atlas Studio.

The Interaction Registry provides a unified mechanism for registering, discovering, and retrieving interaction implementations without coupling consumers to concrete implementations.

The registry acts as the primary extension point for future interaction providers while remaining independent from rendering, document manipulation, and editor business logic.

---

# Goal

Provide a reusable, extensible, and framework-independent registry capable of managing interaction definitions through a stable public API.

---

# Background

Atlas Studio supports many interaction types.

Examples include:

- Pointer Interaction
- Mouse Interaction
- Keyboard Interaction
- Drag Interaction
- Resize Interaction
- Gesture Interaction
- Accessibility Interaction

Rather than hard-coding these interactions into the Interaction Manager, they should be registered through a central registry.

# Responsibilities

The Interaction Registry is responsible for:

- Registering interaction definitions.
- Looking up registered interactions.
- Enumerating available interactions.
- Providing interaction metadata.
- Supporting future interaction providers.

The Interaction Registry is **not** responsible for:

- Executing interactions.
- Managing sessions.
- Routing events.
- Rendering.
- Business logic.

---

# Registry Operations

The registry should support:

- Register
- Unregister
- Lookup
- Contains
- Enumerate
- Clear

Future registry operations should remain backward compatible.

---

# Public API

Example:

```ts
interface InteractionRegistry {
  register(interaction: InteractionRegistration): void;

  unregister(id: string): void;

  contains(id: string): boolean;

  get(id: string): InteractionRegistration | undefined;

  all(): readonly InteractionRegistration[];
}
```

Example registration:

```ts
interface InteractionRegistration {
  readonly id: string;

  readonly type: string;

  readonly name: string;

  readonly description?: string;
}
```

---

# Design Principles

The Interaction Registry should be:

- Framework independent
- Strongly typed
- Extensible
- Immutable
- Lightweight

Interaction implementations should always be discovered through the registry rather than referenced directly.

---

# Technical Requirements

The implementation must:

- Support dynamic registration.
- Preserve immutable registration metadata.
- Avoid framework-specific dependencies.
- Preserve API stability.
- Support future provider extensions.

---

# Accessibility Requirements

The Interaction Registry must:

- Remain independent from presentation.
- Preserve interaction metadata.
- Support accessibility-related interaction providers.
