# TASK-0003 — Implement Interaction State

## Summary

Implement the `InteractionState`, the immutable representation of the current state of an interaction session.

The Interaction State provides a consistent snapshot of an interaction lifecycle that can be safely observed by other packages without exposing mutable implementation details.

The state model serves as the primary communication contract between the interaction layer and consumers such as Atlas UI, Atlas Selection, Atlas History, and Atlas Editor.

---

# Goal

Provide a reusable, immutable, and framework-independent state model that accurately represents the lifecycle of an interaction session.

---

# Background

Every interaction progresses through multiple states during its lifecycle.

Examples include:

- Pointer pressed
- Pointer moving
- Dragging
- Resizing
- Gesture recognition
- Interaction cancelled
- Interaction completed

Instead of exposing mutable internal objects, Atlas Studio should expose immutable interaction state snapshots.

# Responsibilities

The Interaction State is responsible for:

- Representing the current interaction state.
- Providing immutable state snapshots.
- Exposing lifecycle information.
- Carrying interaction metadata.
- Supporting state observation.

The Interaction State is **not** responsible for:

- Managing sessions.
- Routing interactions.
- Rendering.
- Executing commands.
- Business logic.

# Supported States

The state model should support:

- Idle
- Starting
- Active
- Suspended
- Cancelling
- Cancelled
- Completed

Future lifecycle states should remain backward compatible.

---

# Public API

Example:

```ts
interface InteractionState {
  readonly sessionId: string;

  readonly type: string;

  readonly status: InteractionStateType;

  readonly timestamp: number;
}
```

Example lifecycle:

```ts
enum InteractionStateType {
  Idle,

  Starting,

  Active,

  Suspended,

  Cancelling,

  Cancelled,

  Completed,
}
```

---

# Design Principles

The Interaction State should be:

- Immutable
- Serializable
- Strongly typed
- Framework independent
- Observable

State objects should never expose mutable internal implementation.

---

# Technical Requirements

The implementation must:

- Preserve immutable state.
- Support serialization.
- Support future lifecycle extensions.
- Avoid framework-specific dependencies.
- Preserve API stability.

---

# Accessibility Requirements

The Interaction State must:

- Support accessibility workflows.
- Remain presentation independent.
- Preserve interaction semantics.
