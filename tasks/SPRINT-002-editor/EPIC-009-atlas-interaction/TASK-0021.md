# TASK-0021 — Implement Drag Session

## Summary

Implement the `DragSession`, a reusable abstraction responsible for representing an active drag operation within Atlas Studio.

A Drag Session encapsulates the complete lifecycle of a drag interaction, beginning when a drag operation is initiated and ending when it is either completed or cancelled. It provides a unified interaction model that can be reused across editors, panels, workspaces, floating windows, tree views, and plugin-defined components.

The Drag Session is responsible only for representing drag lifecycle information. It does not move objects, update layouts, modify documents, or render drag previews.

---

# Goal

Provide a reusable, extensible, and framework-independent drag session abstraction capable of representing drag operations through a stable public API.

---

# Background

Dragging is one of the most common interactions inside an editor.

Typical drag operations include:

- Moving editor blocks
- Moving floating panels
- Reordering tabs
- Reordering tree nodes
- Moving selections
- Moving widgets
- Plugin-defined drag operations

Instead of allowing every subsystem to implement its own drag lifecycle, Atlas Studio should expose a unified Drag Session abstraction.

# Responsibilities

The Drag Session is responsible for:

- Representing an active drag operation.
- Tracking drag lifecycle.
- Recording drag origin.
- Recording current pointer position.
- Providing immutable drag metadata.

The Drag Session is **not** responsible for:

- Dropping objects.
- Rendering previews.
- Updating layouts.
- Executing commands.
- Business logic.

# Public API

Example:

```ts
interface DragSession {
```

# Design Principles

The Drag Session should be:

- Framework independent
- Immutable where possible
- Strongly typed
- Lightweight
- Extensible

Drag sessions should represent interaction state only and remain independent from rendering or document manipulation.
