# TASK-0002 — Implement History State

## Summary

Implement the `HistoryState`, an immutable abstraction responsible for representing the current state of the history system within Atlas Studio.

The History State provides a canonical representation of history information, including navigation position, history capacity, current entry, and lifecycle status. It serves as the immutable state container used throughout the atlas-history package while remaining independent from rendering, persistence, and editor implementations.

The History State abstraction is responsible only for representing history state. It does not perform history navigation, snapshot management, transaction processing, or business logic.

---

# Goal

Provide a reusable, immutable, and framework-independent history state abstraction capable of representing history information through a stable public API.

---

# Background

A centralized immutable history state simplifies coordination between history recording, undo/redo navigation, diagnostics, and future extensions.

Typical examples include:

- Current history position
- Total history entries
- Current active entry
- Navigation availability
- History capacity
- Lifecycle status

Rather than storing mutable history information across multiple components, Atlas Studio should expose a unified History State abstraction.

# Responsibilities

The History State is responsible for:

- Representing immutable history state.
- Tracking the active history entry.
- Providing navigation metadata.
- Supporting future history extensions.
- Preserving deterministic state transitions.

The History State is **not** responsible for:

- Recording history.
- Restoring history.
- Rendering UI.
- Executing commands.
- Business logic.
