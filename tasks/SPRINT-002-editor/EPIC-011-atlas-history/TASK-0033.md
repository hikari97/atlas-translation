# TASK-0033 — Implement History Dispatcher

## Summary

Implement the `HistoryDispatcher`, the central abstraction responsible for delivering immutable history events to registered listeners throughout Atlas Studio.

The History Dispatcher coordinates event propagation between history producers and history consumers while preserving deterministic delivery order, listener isolation, and immutable event payloads. It serves as the communication hub for the atlas-history event system and remains completely independent from rendering engines, persistence layers, application frameworks, and editor implementations.

The History Dispatcher is responsible only for event dispatching. It does not define event types, manage business logic, mutate history state, or perform rendering.

---

# Goal

Provide a reusable, extensible, and framework-independent dispatcher capable of delivering immutable history events through a stable public API.

---

# Background

An event-driven architecture requires a dispatcher that coordinates communication between publishers and subscribers.

Typical examples include:

- Dispatch history recorded events
- Dispatch undo events
- Dispatch redo events
- Dispatch snapshot events
- Dispatch transaction events
- Dispatch branch events
- Dispatch diagnostic events
- Plugin-generated history events

Rather than coupling history components directly, Atlas Studio should use a centralized History Dispatcher.

# Responsibilities

The History Dispatcher is responsible for:

- Dispatching immutable history events.
- Registering listeners.
- Removing listeners.
- Preserving dispatch order.
- Supporting future dispatch strategies.

The History Dispatcher is **not** responsible for:

- Defining events.
- Recording history.
- Executing business logic.
- Rendering.
- Persistence.
