# TASK-0030 — Implement Selection Cursor

## Summary

Implement the `SelectionCursor`, a reusable abstraction responsible for representing the current navigation position within Atlas Studio.

The Selection Cursor provides a stable navigation reference that can move independently from selection state. It maintains the current navigation position, remembers previous positions, supports traversal strategies, and provides a deterministic navigation context for keyboard navigation, accessibility workflows, and plugin-defined navigation.

The Selection Cursor is responsible only for representing navigation position. It does not render cursor indicators, modify selections, execute commands, or implement business logic.

---

# Goal

Provide a reusable, extensible, and framework-independent Selection Cursor abstraction capable of tracking navigation position through a stable public API.

---

# Background

Most editors internally maintain a navigation cursor even when the user cannot see it.

Typical examples include:

- Keyboard navigation
- Tree navigation
- Layer panel navigation
- Timeline navigation
- Grid navigation
- Canvas navigation
- Plugin-defined cursor navigation

Instead of allowing each subsystem to maintain independent navigation cursors, Atlas Studio should expose a unified Selection Cursor abstraction.

# Responsibilities

The Selection Cursor is responsible for:

- Tracking navigation position.
- Moving between selectable items.
- Remembering previous cursor positions.
- Producing immutable cursor state.
- Supporting future cursor strategies.

The Selection Cursor is **not** responsible for:

- Rendering cursor indicators.
- Selecting objects.
- Updating documents.
- Executing commands.
- Business logic.

# Public API

Example:

```ts

  readonly state: SelectionCursorState;
```

# Design Principles

The Selection Cursor should be:

- Framework independent
- Immutable
- Strongly typed
- Lightweight
- Extensible

Cursor management should remain deterministic and independent from rendering or editor implementations.

---

# Technical Requirements

The implementation must:

- Preserve immutable cursor state.
- Support pluggable traversal providers.
- Avoid framework-specific dependencies.
- Support arbitrary collection sizes.
- Preserve API stability.

---

# Accessibility Requirements

The Selection Cursor must:

- Preserve accessibility navigation semantics.
- Support keyboard-only workflows.
- Support assistive technologies.
- Remain presentation independent.
