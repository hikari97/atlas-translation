# TASK-0016 — Implement Marquee Session

## Summary

Implement the `MarqueeSession`, a reusable abstraction responsible for managing the lifecycle of a marquee selection interaction within Atlas Studio.

A Marquee Session represents a single drag-selection operation that begins when the user initiates a drag gesture and ends when the gesture is completed or cancelled. It maintains immutable session metadata, drag boundaries, interaction state, and selection context while remaining independent from rendering, document storage, and editor-specific behavior.

The Marquee Session is responsible only for representing the state of a marquee interaction. It does not perform hit testing, render selection rectangles, modify documents, or execute application logic.

---

# Goal

Provide a reusable, extensible, and framework-independent Marquee Session abstraction capable of representing drag-selection interactions through a stable public API.

---

# Background

Marquee selection is one of the most common interaction mechanisms in graphical editors.

Typical examples include:

- Drag selection on canvas
- Selecting multiple layers
- Selecting multiple nodes
- Selecting multiple artboards
- Timeline drag selection
- File explorer drag selection
- Plugin-defined marquee workflows

Instead of allowing each subsystem to manage its own drag-selection lifecycle, Atlas Studio should expose a unified Marquee Session abstraction.

# Responsibilities

The Marquee Session is responsible for:

- Representing a marquee interaction session.
- Tracking drag lifecycle.
- Recording immutable session metadata.
- Tracking drag origin and current position.
- Supporting future marquee workflows.

The Marquee Session is **not** responsible for:

- Rendering marquee rectangles.
- Performing hit testing.
- Updating document selections.
- Executing commands.
- Business logic.
