# TASK-0038 — Implement Selection Event Bus

## Summary

Implement the `SelectionEventBus`, a reusable abstraction responsible for publishing and distributing selection-related events throughout Atlas Studio.

The Selection Event Bus provides a centralized, strongly typed event system for broadcasting selection state changes between packages while remaining independent from rendering, document mutation, command execution, and editor-specific implementations.

The Selection Event Bus is responsible only for transporting immutable selection events. It does not modify selection state, execute business logic, or provide persistence.

---

# Goal

Provide a reusable, extensible, and framework-independent event bus capable of distributing selection events through a stable public API.

---

# Background

Large editors require a decoupled communication mechanism between independent modules.

Typical examples include:

- Selection Changed
- Selection Cleared
- Selection Added
- Selection Removed
- Focus Changed
- Range Selection Completed
- Plugin-defined selection events

Instead of tightly coupling packages together, Atlas Studio should expose a unified Selection Event Bus abstraction.

# Responsibilities

The Selection Event Bus is responsible for:

- Publishing selection events.
- Registering listeners.
- Removing listeners.
- Delivering immutable event payloads.
- Preserving deterministic event ordering.

The Selection Event Bus is **not** responsible for:

- Executing commands.
- Rendering UI.
- Persisting events.
- Updating document state.
- Business logic.

# Public API

Example:

```ts
interface SelectionEventBus {
  publish(event: SelectionEvent): void;

  subscribe(listener: SelectionEventListener): void;

  unsubscribe(id: string): void;

  clear(): void;
}
```

Example event:

```ts
interface SelectionEvent {
  readonly id: string;

  readonly type: SelectionEventType;

  readonly timestamp: number;

  readonly payload: unknown;
```

```ts
interface SelectionEventListener {
```
