# TASK-0001 — Define Plugin Contract

## Summary

Define the public Plugin Contract for Atlas Studio.

The Plugin Contract serves as the foundation of the Plugin System by defining the minimum capabilities every plugin must implement.

This contract becomes the primary integration point between `atlas-plugin` and `atlas-core`.

---

# Goal

Create a stable, strongly typed, and extensible plugin contract that can be implemented by both built-in plugins and third-party plugins.

---

# Background

Atlas Studio extends functionality through plugins.

Rather than allowing arbitrary runtime extensions, every plugin must implement a common contract so the runtime can:

- Discover plugins
- Validate plugins
- Initialize plugins
- Manage plugin lifecycle
- Collect metadata

---

# Scope

## Included

- Plugin interface

- Plugin lifecycle contract
- Plugin dependency contract

---

## Excluded

# Responsibilities

The Plugin Contract is responsible for defining:

- Plugin identity
- Plugin metadata
- Plugin lifecycle
- Plugin dependencies
- Runtime integration point

The Plugin Contract is **not** responsible for:

- Loading plugins
- Registering plugins
- Dependency resolution
- Runtime management

---

# Lifecycle

Every plugin follows the lifecycle below.

```text
Install

↓

Initialize

↓

Activate

↓

Running

↓

Deactivate

↓

Dispose
```

---

# Public API

Example:

```ts
export interface Plugin {
    readonly id: string;
    readonly name: string;
    readonly version: string;

    install(context: PluginContext): void | Promise<void>;

    activate(context: PluginContext): void | Promise<void>;

    deactivate(context: PluginContext): void | Promise<void>;
}
```

The public contract may evolve, but it must remain backward compatible whenever possible.

---

# Design Principles

The Plugin Contract should be:

- Strongly typed
- Minimal
- Stable
- Extensible
- Framework independent

Every plugin should expose only the information required by the runtime.

---

# Technical Requirements

The implementation must:

- Avoid runtime-specific APIs.
- Preserve type safety.
- Avoid business logic.
- Support asynchronous lifecycle methods.
- Remain compatible with `atlas-core`.
