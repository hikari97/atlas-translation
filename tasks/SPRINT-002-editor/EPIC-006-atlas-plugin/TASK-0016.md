# TASK-0016 — Integrate Plugin Event System

## Summary

Integrate Atlas Plugin with the Atlas Event System.

This task enables plugins to publish and subscribe to runtime events through the public interfaces provided by `atlas-events`.

The integration must remain loosely coupled and depend only on public contracts exposed by the Event System.

---

# Goal

Allow plugins to participate in the Atlas Event System without introducing direct dependencies on event implementations.

---

# Background

Plugins frequently need to communicate with other runtime components.

Rather than directly invoking each other, plugins should communicate through the Event System.

This approach improves modularity, extensibility, and reduces coupling across the runtime.

---

# Scope

## Included

- Plugin Event Publisher
- Plugin Event Subscriber
- Event Context Integration
- Event Registration
- Event Dispatch Integration

---

## Excluded

- Event Bus implementation
- Event persistence
- Distributed events
- Remote events
- Event replay

# Responsibilities

Plugin Event Integration is responsible for:

- Publishing plugin events.
- Subscribing to runtime events.
- Registering plugin event handlers.
- Connecting plugins with the Event System.

Plugin Event Integration is **not** responsible for:

- Event Bus implementation.
- Event storage.
- Event serialization.
- Event transport.

---

# Integration Flow

```text
Plugin

↓

Plugin Context

↓

Plugin Event Integration

↓

atlas-events

↓

Event Bus
```

---

# Public API

Example:

```ts
interface PluginEventIntegration {
  publish(event: Event): Promise<void>;

  subscribe<T extends Event>(
    type: EventType<T>,
    handler: EventHandler<T>,
  ): void;
}
```

---

# Design Principles

Plugin Event Integration should be:

- Event-driven
- Loosely coupled
- Strongly typed
- Runtime independent
- Easy to test

Plugins must never communicate directly with each other.

---

# Technical Requirements

The implementation must:

- Use only public APIs from atlas-events.
- Avoid circular dependencies.
- Support asynchronous event publishing.
- Support multiple subscribers.
- Preserve public API stability.
