# TASK-0008 — Implement Plugin Discovery

## Summary

Implement the Plugin Discovery component for Atlas Studio.

Plugin Discovery is responsible for locating plugin definitions from registered plugin sources and producing a collection of candidate plugins that can later be validated and registered.

Discovery does not load or execute plugins.

---

# Goal

Provide a consistent mechanism for discovering plugins from multiple plugin sources before runtime initialization.

---

# Background

Applications may obtain plugins from different sources.

Examples include:

- Static registration
- Package manifests
- Workspace configuration
- Local directories
- Custom discovery providers

The Plugin Discovery component normalizes these sources into a single collection of plugin definitions.

# Responsibilities

Plugin Discovery is responsible for:

- Discovering available plugins.
- Reading plugin definitions.
- Supporting multiple discovery sources.
- Preventing duplicate discovery.
- Returning immutable discovery results.

Plugin Discovery is **not** responsible for:

- Registering plugins.
- Loading plugins.
- Validating dependencies.
- Executing plugin lifecycle.

---

# Discovery Flow

```text
Plugin Sources

↓

Discovery Providers

↓

Plugin Definitions

↓

Discovery Result
```

---

# Public API

Example:

```ts
interface PluginDiscovery {
  discover(): Promise<DiscoveryResult>;
}
```

Example provider:

```ts
interface PluginDiscoveryProvider {
  discover(): Promise<readonly PluginDefinition[]>;
}
```

---

# Design Principles

Plugin Discovery should be:

- Deterministic
- Extensible
- Source independent
- Strongly typed
- Immutable

Discovery should not modify plugin definitions.

---

# Technical Requirements

The implementation must:

- Support multiple discovery providers.
- Prevent duplicate plugin identifiers.
- Preserve discovery order.
- Return immutable collections.
- Preserve public API stability.
