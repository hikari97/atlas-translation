# TASK-0010 — Implement Plugin Lifecycle

## Summary

Implement the Plugin Lifecycle subsystem for Atlas Studio.

The Plugin Lifecycle coordinates state transitions of plugins from registration until disposal while ensuring every transition occurs in a predictable and deterministic order.

Lifecycle execution is coordinated by the Plugin Manager but implemented independently through the Plugin Lifecycle component.

---

# Goal

Provide a reusable and deterministic lifecycle implementation that manages plugin state transitions consistently across the runtime.

---

# Background

Every plugin progresses through a series of runtime states.

Without a dedicated lifecycle component, startup and shutdown behavior becomes inconsistent and difficult to maintain.

Separating lifecycle management from the Plugin Manager keeps responsibilities isolated and improves testability.

# Responsibilities

Plugin Lifecycle is responsible for:

- Managing plugin state transitions.
- Executing lifecycle hooks.
- Validating transition order.
- Reporting lifecycle state.
- Publishing lifecycle events.

Plugin Lifecycle is **not** responsible for:

- Discovering plugins.
- Registering plugins.
- Resolving dependencies.
- Bootstrapping runtime.

---

# Lifecycle States

```text
Registered

↓

Installed

↓

Initialized

↓

Activated

↓

Running

↓

Deactivated

↓

Disposed
```

State transitions must always follow the defined order.

---

# Public API

Example:

```ts
interface PluginLifecycle {
  install(plugin: PluginDefinition): Promise<void>;

  initialize(plugin: PluginDefinition): Promise<void>;

  activate(plugin: PluginDefinition): Promise<void>;

  deactivate(plugin: PluginDefinition): Promise<void>;

  dispose(plugin: PluginDefinition): Promise<void>;

  stateOf(pluginId: string): PluginState;
}
```

---

# Lifecycle Rules

- A plugin cannot be activated before initialization.
- A plugin cannot be initialized before installation.
- A disposed plugin cannot be activated again.
- Invalid state transitions must return a lifecycle error.

---

# Design Principles

The Plugin Lifecycle should be:

- Deterministic
- Observable
- Strongly typed
- Independent
- Testable

Lifecycle logic should remain separate from Plugin Manager orchestration.

---

# Technical Requirements

The implementation must:

- Preserve valid state transitions.
- Prevent invalid transitions.
- Publish lifecycle events.
- Support asynchronous lifecycle hooks.
- Preserve public API stability.
