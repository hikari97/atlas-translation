# TASK-0004 — Implement Plugin Context

## Summary

Implement the Plugin Context contract for Atlas Studio.

The Plugin Context provides the runtime environment that plugins interact with during their lifecycle.

It serves as the primary communication boundary between a plugin and the Atlas Runtime while keeping plugins independent from internal runtime implementations.

---

# Goal

Provide a stable and strongly typed Plugin Context that exposes only the runtime capabilities required by plugins.

---

# Background

Plugins should never communicate directly with Runtime Kernel or internal runtime components.

Instead, every plugin receives a Plugin Context that exposes controlled runtime services.

This keeps plugins isolated from runtime implementation details and preserves API stability.

# Responsibilities

Plugin Context is responsible for providing controlled access to:

- Runtime information
- Configuration
- Registered services
- Event publishing
- Command execution

Plugin Context is **not** responsible for:

- Creating services
- Managing runtime lifecycle
- Registering plugins
- Loading plugins

---

# Suggested Structure

Example:

```ts
export interface PluginContext {
  readonly application: Application;

  readonly configuration: Configuration;

  readonly environment: Environment;

  get<T>(token: ServiceIdentifier<T>): T;

  publish(event: Event): Promise<void>;

  execute<TResult>(command: Command<TResult>): Promise<TResult>;
}
```

The exact API may evolve during implementation but should remain minimal and strongly typed.

---

# Design Principles

Plugin Context should be:

- Immutable
- Strongly typed
- Runtime independent
- Minimal
- Stable

Plugins should only access runtime capabilities through this context.

---

# Technical Requirements

The implementation must:

- Avoid runtime implementation details.
- Avoid exposing internal registries.
- Support asynchronous operations.
- Preserve public API stability.
- Remain compatible with atlas-core.
